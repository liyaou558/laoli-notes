## 一、核心观点

**1. 实现本身不再是成本最高的部分——品味、判断力与策展能力正在成为最稀缺的资产。** Andrew Ambrosino 在对话中抛出了一个直击 AI 时代本质的论断：当任何人都能用模型做出任何功能时，技术实现的门槛急剧下降，但"到底应该做什么"这个问题反而变得无比困难。传统产品开发流程建立在"实现很贵"这个核心前提之上——因为写代码、做设计、搭建功能都需要大量时间和人力，所以必须用 PRD、设计评审等流程来降低试错成本。但现在这个前提被彻底瓦解了：模型可以在几分钟内生成一个完整功能的原型，实现本身不再是瓶颈。于是在一个"什么都能做"的世界里，筛选和判断——也就是"品味"——反而成了最值钱的能力。Andrew 强调，品味不只是审美层面的，更包括系统思维：你能否判断一个功能在更大的产品生态中意味着什么？你能否预测用户真正需要什么？你能否辨别什么值得做、什么不值得做？这些判断力无法被模型替代，因为它们源自对人、对市场、对产品的深层理解。

**2. "90 个原型"文化：OpenAI 内部不再写文档，直接上手做。** Andrew 描述了一种与传统产品开发截然不同的工作方式：在 OpenAI，面对一个想法，人们不再花时间写详细的需求文档、画线框图、做竞品分析，而是直接在 Codex 里把东西做出来。他称之为"90 个原型"文化——不是指精确的数量，而是指一种心态：原型极其廉价，所以做足够多的原型直到找到对的方向。这种文化的底层逻辑是：当实现的边际成本趋近于零时，"做出来看看"比"想清楚再做"更高效。传统流程中，PRD 的很大一部分价值在于降低昂贵实现的试错成本；当实现不再昂贵时，直接动手验证比纸面推演更有说服力。但 Andrew 也指出一个微妙之处：原型有时反而是错误的第一步——因为当你飞快地做出了一个看起来不错的东西，你可能被它的"完成感"欺骗，误以为自己找对了方向，而忽略了选题本身可能就是错的。这就是为什么他反复强调"原始笔触"的概念：有时最粗糙的思考——在没有被任何实现美化之前——反而更接近真相。

**3. PRD 没有死——但选对表达观点的媒介变得极其关键。** 针对"PRD 已死"这个流行观点，Andrew 给出了一个更具辩证色彩的回答。他不同意 PRD 已死，但他认为问题的关键不在于写不写 PRD，而在于你选择的表达媒介是否匹配你的目标。如果实现变得极其充裕，那么"选对表达观点的形式"就成了一个战略级决策：有些想法最适合用文字文档来表达，有些最适合用原型来展示，有些则最适合在白板上画出来。把产品流程跟某一种特定媒介绑定——比如"必须先写 PRD 再画线框图再出高保真设计"——才是真正危险的事情。他特别批评了传统设计流程中的"案例研究工厂"问题：设计师花大量时间制作精美的案例研究，但这些产出本身并没有推进对产品的理解。而现在的情况是，你可以把整个实现都拉进流程——在真实的、可交互的产品中讨论想法，而不是在抽象的文档中讨论想法。

**4. 角色边界正在消融，但消除角色概念本身可能是危险的。** Codex 团队的角色崩塌比 OpenAI 其他部门更加明显。Andrew 描述了一个真实状态：在团队中，谁是 PM、谁是设计师、谁是工程师的界限越来越模糊，因为每个人都可以借助 Codex 跨越自己的传统边界。他提出了一个富有洞察力的定义——"你的角色是你花时间所做事情的平均值"。团队内部甚至开始使用"技术团队成员"这个统一的称呼，而非区分具体职能。但 Andrew 同时警告：完全消除角色概念可能危险地抹杀专业领域和最佳实践。他明确表示"取消产品角色是个非常糟糕的主意"，因为虽然工具让跨界变得容易，但真正的专业判断力——比如一个资深 PM 对用户需求的直觉、一个资深设计师对交互模式的经验——是无法通过工具快速获得的。Codex 团队的构成反映了这种张力：工程师两位数，设计师大概一半人数，产品工作像"区域防守"一样被全团队分担。最有价值的人不是某种特定角色，而是那种"能把想法从概念做到完成，并且有品味判断好坏"的人。

**5. 去做那些现在还不行、但等模型变强之后就能行的东西。** Andrew 分享了一个关于时机判断的深刻洞见。他以 Codex 桌面应用为例：如果 OpenAI 在十一月发布会上推出，它会失败；但等到二月再推出，它就成功了。区别不在于产品本身——产品几乎没有变——而在于底层模型的能力在短短三个月内跃升了一个台阶。这揭示了一种新的产品策略：不要固执地认为跑不通就是烂功能，可能只是还没到时候。你需要有判断力去区分"方向错了的烂功能"和"方向对但模型还没跟上的前瞻性功能"。在这个模型能力飞速进化的时代，推动团队更有野心变得重要——因为做难事变得容易太多了，今天看起来离谱的想法，可能下个季度就变得可行。Andrew 自己的工作方式也体现了这一点：他的个人目标是把 Codex 做成自己写代码用的工具，这意味着他始终在产品中亲自使用，感知模型的进步节奏，从而做出更精准的时机判断。

**6. Codex 正从开发者工具演变为通用知识工作平台——这是 dogfooding 发现的意外真相。** Andrew 揭示了一个 OpenAI 内部的关键发现：当团队 dogfooding（内部使用）Codex 时，他们发现非工程师也在疯狂使用它——整理电脑文件、起草文档、分析数据、阅读邮件。更惊人的是，一旦进入 Codex 应用，没有人愿意离开它去用其他应用。这彻底改变了 Codex 的产品定位：它不再只是一个写代码的工具，而是一个"大本营"——用户在这里调用任何需要的工具和能力，无论是写代码、写文档、处理数据还是自动操作电脑。Andrew 描述这个演变是"曲折的"——Codex 最初是命令行工具，后来变成桌面应用，而每一次转变都源于观察用户实际在用它做什么，而不是提前规划好的路线图。现在 Codex 的愿景是成为一个通用的知识工作界面，用户在其中完成各种工作，Codex 负责调用底层的任何工具和模型。开发者工具和通用知识工作工具的界限"不是非此即彼"——它是一个光谱，而 Codex 正在从一端向另一端快速移动。

**7. 在 AI 时代成功，需要的自我认知程度高得离谱。** Andrew 在对话尾声抛出了一个令人深思的观点：AI 时代对个人自我认知的要求远超以往。因为当你能够借助 AI 做任何事情时，"你应该做什么"就完全取决于你对自己的了解——你知道自己擅长什么、喜欢什么、独特能交付的成果是什么。他给出一条实用建议："别死守你现在的流程，要死守的是你独特能交付的成果。"流程会变、工具会变、角色的边界会变，但如果你清晰地知道自己的独特价值是什么，就能够在变化中找到自己的位置。Andrew 自己的职业经历就是最好的注脚：从设计师转型工程师，再转型产品经理，他称之为"给自己的职业找到了产品契合度"。他同时坦承，讨厌重复的人当创始人挺惨的——因为作为创始人，你必须不断重复愿景、重复价值观、重复方向，这是无法被 AI 替代的工作。

## 二、关键问答

**Q: AI 如何颠覆了传统的产品开发流程？**

A: 传统产品开发流程建立在一个核心前提之上：实现很贵。写代码需要工程师的时间，做设计需要设计师的时间，因此必须在动手之前通过 PRD、评审、规划等流程来降低试错成本。但现在这个前提被 AI 彻底瓦解了——任何人都能用模型在几分钟内做出一个完整功能的原型。这意味着流程发生了"倒置"：过去是"先想清楚再做"，现在是"先做出来再看"。在 OpenAI，面对一个想法，人们不再写文档，而是直接在 Codex 里把东西做出来——这就是"90 个原型"文化的来源。当实现的边际成本趋近于零时，直接动手验证比纸面推演更高效。但 Andrew 同时指出，这并不意味着流程完全无用——选对表达观点的媒介仍然是关键决策。

**Q: 为什么说"品味"比"实现"更值钱？**

A: Andrew 的核心逻辑是：在一个"什么都能做"的世界里，稀缺的不再是"能不能做出来"，而是"到底应该做什么"。品味的含义远超审美——它包括系统思维（判断一个功能在更大生态中的位置）、用户直觉（预测用户真正需要什么）、以及策展能力（在无数可能性中选择最值得投入的方向）。他直言："实现本身不再是成本最高的部分了。我斗胆说一句，是品味。是筛选的过程。"这同时也是一种危险：因为当你能快速做出任何东西时，你也很容易被表面的"完成感"欺骗，误以为自己做对了方向。所以品味的另一层含义是"知道什么时候原型反而是错误的第一步"。

**Q: PRD 真的死了吗？Andrew 怎么看？**

A: Andrew 不同意"PRD 已死"这个简单化的结论。他认为真正的问题不是写不写 PRD，而是你是否选对了表达观点的媒介。在 AI 时代，"如果实现变得极其充裕，那选对表达观点的形式就变得非常重要"。有些想法最适合用文字文档表达，有些最适合用可交互原型展示，有些则在白板上画出来更有效。把设计流程跟某种特定媒介绑定——比如必须先写 PRD 再画线框图再出高保真设计——才是真正危险的地方。他还特别批评了传统设计流程的"案例研究工厂"问题：花大量时间制作精美案例研究，但这些产出并没有推进对产品的真正理解。

**Q: Codex 团队的角色崩塌具体是什么样的？**

A: Andrew 描述了一个比 OpenAI 其他部门更明显的角色融合状态。在 Codex 团队中，PM、设计师、工程师的界限越来越模糊——因为每个人都能借助 Codex 跨越传统边界。团队内部使用"技术团队成员"这个统一称呼。最有价值的人不是某种特定角色，而是那种"能把想法从概念做到完成，并且有品味判断好坏"的人。但 Andrew 同时警告：完全消除角色概念可能危险地抹杀专业领域和最佳实践。他明确表示"取消产品角色是个非常糟糕的主意"，因为虽然工具让跨界变得容易，但真正的专业判断力——比如资深 PM 对用户需求的直觉——无法通过工具快速获得。目前 Codex 团队中工程师约两位数，设计师约一半人数，产品工作像"区域防守"一样被全团队分担。

**Q: "90 个原型"文化具体是指什么？**

A: 这不是一个精确的数字，而是一种工作心态：当原型的制作成本极低时（在 Codex 中几分钟就能做一个），你应该做足够多的原型直到找到对的方向，而不是花大量时间在纸面上推演。在 OpenAI 内部，面对一个新想法，人们不再先写文档和做竞品分析，而是直接在 Codex 里把东西做出来看看。这种文化的前提是 AI 让实现的边际成本趋近于零。但 Andrew 也提出了一个微妙的警告：原型有时反而是错误的第一步——因为当你飞快地做出了一个看起来不错的东西，你可能被它的"完成感"欺骗，误以为自己找对了方向，而忽略了选题本身可能就有问题。他称这种粗糙的、未被实现美化的初始思考为"原始笔触"，认为它有时比精美的原型更接近真相。

**Q: 在模型能力快速变化的情况下，怎么做路线图和规划？**

A: Andrew 给出的原则是"越短期越需要细节"。在模型能力每几个月就跃升一个台阶的环境下，长期路线图变得极其困难——你根本不知道三个月后模型能做到什么。他以 Codex 桌面应用为例：同样的产品在十一月推出会失败，二月推出就成功了，区别仅仅在于底层模型的能力在这三个月内跃升了。这引出了一个重要的策略思维：不要固执地认为跑不通就是烂功能，可能只是还没到时候。好的产品判断力需要区分"方向错了的烂功能"和"方向对但模型还没跟上的前瞻性功能"。Andrew 还强调，在这种环境下要"推动团队更有野心，因为做难事变得容易太多了"——今天看起来离谱的想法，可能下个季度就变得完全可行。

**Q: Andrew 自己是如何使用 Codex 的？**

A: Andrew 自称"拥有全世界最棒的工作"——他的个人目标就是把 Codex 做成自己写代码用的工具，这意味着他每天都在产品中亲自使用、切身感受模型的进步节奏。他分享了一个具体的使用场景：每日简报功能，可以汇总 3000 个 Slack 频道的信息，让他在短时间内掌握团队动态。他还频繁使用 Codex 设置各种自动化——而这个过程本身就是产品发现，因为在自动化自己的工作时，他会自然发现产品需要改进的地方。他还提到了"computer use"能力——让 AI 直接接管电脑操作——这代表了 Codex 从被动工具向主动代理的演进方向。

**Q: Codex 从开发者工具变成通用知识工作平台，这个转变是怎么发生的？**

A: 这个转变主要源于内部 dogfooding 的意外发现。当 OpenAI 团队大规模内部使用 Codex 时，他们发现非工程师——包括运营、商务、管理人员——也在疯狂用它来整理电脑文件、起草文档、分析数据、阅读邮件。更关键的行为信号是：一旦用户进入 Codex 应用，没有人愿意离开它去用其他应用。这让团队意识到，Codex 的定位不应该被限定为"代码工具"，而应该是一个"大本营"——用户在其中调用任何需要的工具和能力。Andrew 强调，开发者工具和通用知识工作工具的界限"不是非此即彼"，Codex 正在从命令行工具演变为桌面应用，这个路径是曲折的——每一次产品形态的变化都源于观察用户实际行为，而非提前规划的路线图。

**Q: Andrew 的职业转型经历有什么启示？**

A: Andrew 的职业路径非常独特：从设计师转型工程师，再转型产品经理。他称这个过程为"给自己的职业找到了产品契合度"——就像产品需要找到 product-market fit 一样，个人也需要找到自己最擅长的角色和领域。这个经历让他天然理解角色之间不应该有硬边界。但他也给出了一个微妙的警告：讨厌重复的人当创始人挺惨的，"因为你必须重复"——重复愿景、重复价值观、重复方向，这种重复是创始人的核心工作，无法被 AI 替代。他的职业转型也印证了他的核心观点：在 AI 时代，不要死守你现在的流程和角色标签，要死守的是你独特能交付的成果。

**Q: 在 AI 时代成功，最需要什么素质？**

A: Andrew 的回答是"自我认知——程度高得离谱"。当 AI 让你几乎可以做任何事情时，"你应该做什么"就变成了最核心的问题，而回答这个问题需要你对自己有极深的了解：你擅长什么？你喜欢什么？你独特能贡献的价值是什么？他给出的实用原则是："别死守你现在的流程，要死守的是你独特能交付的成果。"流程会变、工具会变、角色边界会变，但如果你清楚地知道自己的独特价值，就能够在任何变化中找到位置。他还认为，优秀的判断力——知道什么是好的、什么值得做——在 AI 时代变得前所未有的重要，因为它无法被模型替代。
