# 📄 精读：如何把 Codex 用到更深一层

**oneliner**：如果你在用 Coding Agent 但总觉得它只能在仓库里改 diff，这篇文章讲的是怎么把 Agent 当"数字同事"来设计——不只是让它写代码，而是让它能看页面、等事件、保存现场、定时回来继续，形成一个跨代码、页面、消息、时间和人的持续工作流。

## 一、核心论点：Codex 从"写代码工具"到"工作流运行时"

文章用一个很具体的场景开场——让 Agent 修一个前端问题。这个任务天然跨多个域：读代码 → 改代码 → 打开页面看效果 → 需要登录态 → 跑测试 → 等 PR review → 第二天回来处理反馈 → 把进展摘要和未解决问题留给下一个接手的人。

这件事的核心矛盾在于：**一段工作跨过代码、页面、消息、文档、时间和人类判断以后，怎么还能保持连续？**

若飞的核心判断是：Codex 的新能力（durable threads、side panel、browser、automations、goals、memory、skills）不能孤立地当功能点来读。把它们放到同一个工程现场里，拼起来的是另一件事——**一个围绕代码周边工作的轻量工作流运行时**。

这个判断在架构层面的分量很重。它意味着 Codex 的竞争维度正在从"谁的模型代码写得更好"转向"谁能让 Agent 在真实工作环境中持续地、可靠地把事情做完"。前者是模型能力，后者是运行时设计。

---

## 二、论据1：三个边界都在往外扩

### 工作边界：从 repo 到整台电脑

传统 Coding Agent 的活动范围是：仓库文件、shell 命令、测试框架、diff。但真实工作中大量动作发生在这些之外——打开本地页面调 UI、操作需要登录态的内部工具、导出文档、响应消息事件。

Codex 提供了三层触达：

| 触达层 | 适用场景 | 主要风险 |
|--------|---------|---------|
| In-app Browser | 本地页面、公开页面、可视化验证 | 页面内容不可信 |
| Chrome Extension | Gmail、Salesforce 等需要登录态的站点 | cookie/凭据泄露 |
| Computer Use | macOS 桌面应用、GUI 操作 | 误操作不可逆 |

这三层的设计不是功能堆砌，而是对"工作现场"范围的重新定义。Agent 能碰的东西越多，系统需要约束的东西也越多——而约束本身就是 Harness 的核心职责。

### 时间边界：从同步到跨越时间

Automations 解决了"什么时候回来"的问题，不只是定时触发，还包括事件驱动——PR 有新评论、CI 变红、issue 状态变化时自动唤醒 Agent。这意味着 Codex 的工作模式从"你叫它、它干活"变成了"它自己知道什么时候该回来看看"。

跨时间工作的关键是**现场可恢复**。thread 保存现场不只是保存聊天记录，更重要的是保存当时的文件状态、产物、未解决问题——这就是"工作集"的概念。不是所有上下文都要保留，而是整理过的、能让任务继续往下走的那些东西。

### 角色边界：从一个人到多人协作

Side Panel 和 Artifacts 的设计意图不只是"让用户看到结果"，更关键的是**给人一个审查面**。当 Agent 持续做事时，不同角色需要在不同时间点介入：reviewer 要看改动摘要和截图，PM 要看进度，接手的人要能快速理解当前状态。

这不只是 UI 问题，是工作流的分工问题。

## 三、论据2：Memory 和 Skills 的工程语义

文章对这两个能力做了很清晰的区分：

- **Memory**：管的是"过去哪些信息可以影响未来"。不是把所有聊天记录都存下来，而是筛选出那些在后续任务中真正会改变决策的信息。坏记忆会污染新任务，好记忆是跨任务的杠杆。
- **Skills**：管的是"这类事情以后按什么流程做"。它和 prompt 模板的区别在于，Skills 是**可验证、可版本化、可组合的流程片段**，而不是一段一次性提示词。

这个区分在工程上很重要。Memory 是**经验层**（过去任务产生的数据如何指导未来），Skills 是**流程层**（可复用的操作模式如何沉淀）。两层共同解决了"Agent 怎么越用越好"的问题——Memory 让它记住教训，Skills 让它固化最佳实践。

## 四、论据3：最小可落地的入口

文章后半部分给了一个非常务实的最小启动方案：先写三份小文件。

**THREAD.md**（线程描述）：
- 这不是 README，也不是设计文档
- 内容是：这个线程围绕什么目标、当前在做什么、关键文件有哪些、待解决问题、下一个接手的人需要知道什么
- 作用是让下次恢复现场时有据可查

**GOAL.md**（目标定义）：
- 定义成功和停止条件——什么算完成、什么算失败、什么时候该停下来让人介入
- 长任务最大的风险不是模型犯错，而是它不知道什么时候该停
- 目标定义越具体，Agent 越不容易走偏（这也呼应了 /goal 命令的设计意图）

**PERMISSIONS.md**（权限边界）：
- 明确哪些操作 Agent 可以自主执行、哪些需要审批、哪些绝对禁止
- 到"Computer Use 能碰桌面应用"这一层时，权限边界就不再是"nice to have"，而是安全底线

这三份文件构成了一个最小可运行的"工作现场 description"。它们是 Harness（模型之外那圈约束系统）的最简实现——不需要复杂框架，从文件约定开始。

## 五、深度思考

### 正面验证：Agent 工程的真正瓶颈已经从模型能力转移到了运行时

文章的判断和当前行业趋势高度吻合。过去两年 Coding Agent 的核心差异化在模型能力（谁更会写代码），但 2026 年后模型能力趋于收敛，真正拉开差距的是**Agent 怎么在真实工作流里持续运作**——上下文管理、工具编排、权限控制、状态持久化、跨时间恢复。

这一点在 Hermes Agent 的实践中也能验证。我们在启动 Hermes 时，Memory、Skills、cron 这些"运行时设施"对实际效用的贡献，经常大于模型本身的版本迭代。Codex 的 durable threads + automations + side panel 也是在回答同一类问题：Agent 不是在"一次对话"里完成工作的，而是在"一段持续的过程中"完成工作的。

### 反驳与盲区：如果工作现场就是不可恢复的呢？

文章假设了一个重要前提：工作现场是**可描述、可恢复**的。但在真实工程中，很多状态根本无法用 THREAD.md 完整表达——比如某个 bug 的定位过程涉及十几次试错性的代码改动和回滚，每次试错都在模型上下文和本地文件系统中留下了痕迹，但这些痕迹之间的关系很难被结构化地"写下来"。

另一个盲区是**多人并发**。文章讨论了"下一个接手的人"，但假设的是串行交接。如果两个 Agent 同时在一个 repo 里工作（一个修 bug、一个做 feature），THREAD.md 的工作集模式会出现冲突。文章把这归到"需要更好工具"的类别里，但跳过了一个根本问题：Agent 之间的协作协议到底是什么？

### 意外延伸：Side Panel 其实是在给 Agent 做"可解释性 UI"

文章提到 Side Panel 是用来展示"产物和推理步骤"让人类审查。但从 AI 工程化的角度看，这实际上是在给 Agent 做一个**可解释性界面**。传统 LLM 的可解释性是 attention 可视化、logit lens 这类学术方法，而 Side Panel 走的是完全不同的路线——不是解释模型内部，而是解释 Agent 的外部行为链。这种"行为可解释性"对于工程落地来说，可能比模型内部可解释性更有价值。

---

## 结论

这篇文章的核心价值不在功能介绍，而在把一组新能力串成了一个完整的工程视角：**Agent 已经从"写代码的模型"变成了"需要运行时的系统"**。对应到架构思路上，要做的是三件事——定义工作现场（THREAD.md）、设定边界和停止条件（GOAL.md + PERMISSIONS.md）、搭建触达层和状态持久化（Browser/Chrome/Computer Use + Memory/Skills）。

最小启动的路径也很清楚：从三份小文件开始，而不是等一个完整的框架。
