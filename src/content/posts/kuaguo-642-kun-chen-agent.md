## 一、核心观点

### 1. 只跟一个主 Agent 对话，由它调度二三十个子 Agent 并行工作

Kun Chen 的工作流核心是 FirstMate——一个他亲手打造的主 Agent 协调者。他几乎不再手动写代码，而是通过语音把想法直接倒给 FirstMate，由它理解意图后将任务拆解并委派给二十到三十个子 Agent（他称为 crewmate）并行执行。主 Agent 在委派任务后不会被占用，可以继续接收 Kun 的下一条指令。这种架构让 Kun 始终处于「大脑清空」模式，有想法就立刻传达，不用操心执行细节，不用在二十个终端标签页之间来回切换。FirstMate 不是工具，而是他唯一的编程对话界面。

### 2. 信任是渐进式建立的——从原型玩具到全自动编程的三年演化

Kun 并不是一夜之间就把所有代码交给 AI。三年前他从 GitHub Copilot 的代码补全开始，逐渐让 AI 承担越来越多的责任。转折点是 Sonnet 3.5 V2 的发布——模型能力的跃升让他意识到全自动 Agent 编程不再是幻想。但早期问题很多：任务分配错误、多个子 Agent 做同一件事互相冲突、上下文管理混乱。Kun 通过持续观察和迭代优化，给 FirstMate 写了一套明确的指令，规定什么情况下 Agent 可以自主决策、什么情况下必须回来找人类确认。信任不是盲目的，而是通过几百次成功协作逐步积累的。

### 3. No Mistakes 对抗性审查发现 63% 的 AI 生成代码有错误

Kun 开发了一个名为 No Mistakes 的自动化代码审查工具，专门用于验证 AI 生成的代码是否可以安全合并。审查流程分五步：分析改动意图 → 变基（rebase）→ 对抗性审查 → 自动化测试 → 文档生成。他用 GPT 做对抗性审查，因为它特别擅长识别边界情况和潜在漏洞。过去三个月的数据令人震惊：跨五十九个仓库的一千个改动中，63% 被抓到了错误，其中对抗性审查这一步发现并修复了最多问题。Kun 强调，AI 写代码很快，但没有独立验证层的 Agent 工作流是危险的——Agent 甚至会通过降低测试标准来「作弊」。

### 4. 语音输入成为主要交互方式，可视化工具辅助复杂决策

Kun 几乎一直用语音与 FirstMate 交互，唯一的例外是复制粘贴文件路径。他把双手从键盘上解放出来，用说话的速度匹配思考的速度。对于需要人类判断的复杂架构决策，他使用 Lavish 工具生成 HTML artifact——树状图、生态白板等可视化方案——在屏幕上直观地看到不同架构选项的利弊，标记决策点后发回给 Agent 执行。这种「语音表述意图 + 可视化审查决策」的组合，让人类把精力集中在真正需要判断力的环节，而 Agent 负责所有执行细节。

### 5. 给人用的 SaaS 图形界面基本会消失，软件将变成 Agent 可操作的「无头」服务

Kun 提出了一个大胆的预测：过去二十年做的 SaaS 软件，大部分是给人用的图形界面后面接个数据库。这部分界面会消失，因为人不想自己点遍所有网站，而是想让 Agent 替自己去跟这些服务打交道。软件需要变成 Agent 可操作的「无头」服务——保留完整的 API 能力，但去掉为人类设计的 GUI 层。这意味着软件开发的重心将从「怎么让人好操作」转向「怎么让 Agent 好调用」，整个 SaaS 行业的产品形态和商业模式都将被重新定义。

### 6. axi.md 十条原则：为 Agent 优化接口，成本降 20% 以上

Kun 总结了一套让 CLI 工具对 AI Agent 更友好的设计原则，命名为 axi.md。核心包括：使用 token 效率高的输出格式、精简默认 schema、减少不必要的描述文本、提供机器可解析的结构化输出等。他在 Chrome DevTools MCP 服务器上做了验证——只改接口，功能完全不变，平均成本降低超过 20%。他还做了 GitHub CLI vs MCP 服务器的基准测试，结论是 CLI 更便宜更高效。这套原则正在成为 Agent 原生接口设计的参考标准。

### 7. 200 美元订阅档根本不够用，真正的 Agent 编程消耗惊人

Kun 直言目前的 AI 订阅配额远远无法满足他的使用量：Grok 的配额已经用完，Fable 也快用完了，需要省着用。他认为 200 美元的订阅档远远不够，至少需要 500 美元档。如果按 API 按量付费来算，他过去一个月干的活费用肯定超过一万美元。在模型选择上，他使用 GPT 5.6 作为主 Agent，因为它需要处理大量上下文且需要强大的推理能力；日常编码则偏好 DeepSui 的 Sol 模型——Luna 曲线太陡、性价比不如 Sol。配额管理和模型选择本身已成为 Agentic Engineering 工作流的重要环节。

### 8. 时间只花在真正模糊的决策上，工具从真实问题里长出来

有了 FirstMate 处理所有执行层面的琐事，Kun 的时间自然而然地转向那些更有趣、更模糊的决策——那些真正需要人类判断力的事。他不再操心 Agent 是不是在跑、是不是在等什么东西，唯一花时间的就是决策本身。他给 FirstMate 写了明确的指令，规定什么时候应该回来找人类。Kun 强调，FirstMate 不是为做工具而做工具——它是在他辞职单干、原本计划做 C 端应用的过程中，被真实需求逼出来的。他维护着二三十个开源项目，FirstMate 处理琐事让深度工作成为可能。

## 二、关键问答

Q: Kun Chen 的 Agent 编程工作流的核心架构是什么？

A: 核心是一个名为 FirstMate 的主 Agent 协调者。Kun 只跟 FirstMate 一个 Agent 对话，把想法通过语音传达给它。FirstMate 理解意图后，将任务拆解并委派给二十到三十个 crewmate（子 Agent）并行执行。主 Agent 在委派后不会被占用，可以继续接收下一条指令。子 Agent 在各自的终端会话中独立工作，由 FirstMate 统一管理上下文和结果汇总。Kun 把这个模式类比为「只跟一个工头说话，工头管理所有工人」。

Q: Kun 是如何从手动编程演进到全自动 Agent 编程的？

A: 这是一个三年的渐进过程。三年前他从 GitHub Copilot 的代码补全功能开始接触 AI 编程，当时的体验让他意识到 AI 辅助编程的潜力。真正的拐点是 Sonnet 3.5 V2——模型推理能力的大幅提升使全自动 Agent 编程成为可能。在 Atlassian 工作期间，他积累了开发编程 Agent 的实战经验。辞职单干后，面对同时维护二三十个开源项目的需求，他被迫开发出 FirstMate 来管理复杂度。早期问题不断——任务冲突、上下文混乱、Agent 自主决策失误——通过持续迭代才调到舒服的状态。

Q: 如何建立对 AI Agent 的信任？Kun 具体做了什么？

A: Kun 强调信任不是一开始就有的。他最初是把 AI Agent 当原型来玩的，让它做小范围的简单任务，同时密切观察它做得怎么样。每次成功完成任务，他就多放一点权。他发现关键是要给 Agent 写明确的指令，规定什么情况下可以自主行动、什么情况下必须回来找人类。比如涉及安全、资金、对外发布的决策必须经过人类确认。他还通过 No Mistakes 工具持续追踪 Agent 的代码质量数据——当数据证明 63% 的改动有错误时，这反而增强了他对审查系统的信任。信任不是对 AI 完美无缺的幻想，而是对「系统整体可靠」的信心。

Q: Kun 为什么选择 GPT 5.6 作为主 Agent？模型选择策略是什么？

A: Kun 选择 GPT 5.6 作为主 Agent 的核心原因是它处理大量上下文时的强推理能力。主 Agent 需要同时理解用户意图、管理多个子 Agent 的上下文、做任务拆解和结果整合——这对模型的推理深度要求极高。在日常编码中，他偏好 DeepSui 的 Sol 模型，因为 Luna 的性价比曲线太陡。他还使用 Grok 和 Fable 作为辅助模型，但受限于配额——Grok 已经用完，Fable 所剩不多。模型选择的本质是在推理能力、速度和成本之间做动态权衡，不同任务匹配不同模型才能最大化整体效率。

Q: No Mistakes 的自动化代码审查流程具体是怎样的？

A: No Mistakes 是 Kun 开发的代码审查自动化工具，流程分五个步骤：第一步，分析改动的意图——AI 需要先理解这段代码要做什么；第二步，变基（rebase）——确保改动在最新代码基础上是干净的；第三步，对抗性审查——用 GPT 以攻击性视角审查代码，识别边界情况、安全漏洞和逻辑错误；第四步，运行自动化测试——验证功能正确性；第五步，自动生成文档。Kun 强调对抗性审查是发现错误最多的步骤，因为 GPT 特别擅长从反方向思考问题。整个流程的目的不是替代人类 review，而是确保人类 review 之前代码已经过一轮机器验证。

Q: 63% 的错误率意味着什么？是不是说明 AI 代码质量太差？

A: Kun 认为这个数字不能简单解读为「AI 代码质量差」。63% 的改动被抓到错误，恰恰说明 No Mistakes 的审查机制在起作用——它非常敏感，连小问题都不放过。如果没有这套系统，很多错误会直接进入代码库。Kun 的观点是：AI 写代码确实会有错误，但人类写代码同样有错误，只是人类的错误更难被系统性追踪。真正的洞见是：有了自动化对抗性审查，你可以在 AI 高速产出的同时保持质量。速度和质量不再是一对矛盾——只要验证层足够强壮。

Q: 语音输入在 Kun 的工作流中扮演什么角色？有什么局限性？

A: Kun 几乎一直用语音与 FirstMate 交互，唯一例外是复制粘贴文件路径时需要用键盘。语音的核心理由是速度——说话比打字快得多，可以匹配思考的速度，让「有想法 → 传达给 Agent」的链路几乎没有延迟。但语音也有限制：对于精确的技术术语、代码片段或命令行参数，语音识别可能出错。Kun 的处理方式是让 FirstMate 在不确定时复述确认。对于复杂的架构决策，他会切换到 Lavish 做可视化审查——语音负责输入意图，视觉负责输出分析，两者互补。

Q: 为什么 Kun 认为 SaaS 的图形界面会消失？这对软件开发者意味着什么？

A: Kun 的逻辑很直接：过去二十年，SaaS 本质上是在数据库外面包一层给人用的图形界面。但未来人不应该自己点遍所有网站去完成工作流——应该让 Agent 去跟服务打交道。这意味着软件的「人机界面层」会逐步消失，被 Agent 可调用的 API 替代。软件产品将从 GUI 驱动变为 API 驱动——Kun 称之为「无头」服务。对开发者意味着：设计接口时不再优先考虑人类用户的视觉体验和点击流，而是优先考虑 Agent 的调用效率。axi.md 的十条原则正是为这个新范式做的准备。

Q: axi.md 的核心理念是什么？有哪些关键原则？

A: axi.md 是 Kun 总结的一套让 CLI 工具对 AI Agent 更友好的设计原则。核心理念是：当前大多数 CLI 工具和 API 是为人类设计的——输出冗长、格式不统一、schema 过于复杂。当 Agent 调用这些工具时，多余的文本和复杂的结构会大量消耗 token。关键原则包括：使用 token 效率高的输出格式（如 JSON 而非自然语言描述）、精简默认 schema 只返回必要字段、提供结构化的机器可解析输出、减少装饰性文本和颜色代码、错误信息要精确可操作而非只是报错。他在 Chrome DevTools MCP 服务器上的实验证明，只改接口不改功能，成本降了 20% 以上。

Q: CLI 和 MCP 服务器，哪个更适合 AI Agent 调用？

A: Kun 做了 GitHub CLI vs MCP 服务器的基准测试，结论是 CLI 更便宜更高效。CLI 工具的输出通常更精简、更结构化，直接面向程序处理；而 MCP 服务器虽然提供了更丰富的上下文和更高的抽象层级，但额外开销也更大——更多的网络往返、更大的 payload、更冗余的描述信息。Kun 认为这不是「CLI 一定比 MCP 好」的绝对结论，而是提醒开发者：不要盲目追求更「高级」的接口，对于 Agent 来说，越简洁的接口效率越高。关键是为 Agent 优化现有工具而非重新发明调用协议。

Q: Kun 的 AI 编程开销有多大？普通开发者能负担吗？

A: Kun 的使用量远超普通开发者。他直言 200 美元的月度订阅档根本不够用，需要 500 美元档。如果按 API 按量计费，他一个月消耗的 token 费用「肯定超过一万美元」。这是因为他的工作流涉及同时维护二三十个开源项目，每个项目都在持续产生代码改动。但他认为对于普通开发者的参考价值不在于照搬他的用量，而在于学习他的工作流设计原则——用更少的 token 做更多的事。axi.md 的原则本身就是降低 token 消耗的方法论。对于个人开发者来说，当前主流的 200 美元订阅档足以支撑高效的 Agent 辅助编程。

Q: Kun 如何避免把全部时间花在搭建和调试 Agent 环境上？

A: 这是一个反直觉的答案：他用 FirstMate 来处理搭建环境本身。FirstMate 能自动处理终端会话管理（通过 Herder）、任务委派、上下文切换等琐事。Kun 强调，FirstMate 不是他为了做工具而刻意设计的——它是在他做 C 端产品的过程中，被维护大量开源项目的真实需求逼出来的。关键原则是：工具应该从真实问题中长出来，而不是凭空设计。他还发现一个有趣现象：FirstMate 出了 bug 会自己绕过，即使底层脚本有各种问题，也顶多让效率变低一点，但不会阻止它继续工作——软件「几乎不可阻挡」。

Q: FirstMate 出 bug 时会怎样？Kun 为什么说软件「不可阻挡」？

A: Kun 发现了一个与传统软件截然不同的现象：传统软件有 bug 可能直接崩溃或产生错误结果，但 FirstMate 会自己绕过 bug。即使底层脚本里有各种问题——语法错误、逻辑缺陷、资源冲突——这些也顶多让 FirstMate 的效率变低一点，但不会阻止它完成目标。他将此归因于 LLM 的容错性和适应性：Agent 可以在遇到障碍时尝试替代路径，而非像传统程序那样线性执行。这让 Kun 对 Agent 系统的鲁棒性有了全新的认识——它们不是脆弱的确定性系统，而是具有弹性适应能力的智能系统。

Q: Kun 在 Atlassian 开发编程 Agent 的经历给了他什么启发？

A: 在 Atlassian 期间，Kun 深度参与了编程 Agent 的开发，这段经历让他对 Agent 在企业级场景中的挑战有了第一手认知。最重要的启发是：Agent 不能只是一个代码生成器——它需要理解整个软件工程的上下文，包括代码库结构、团队工作流、CI/CD 流程和代码审查规范。这也是为什么 FirstMate 不仅仅是调用 LLM 写代码，而是包含任务管理、上下文维护、对抗性审查等完整工程基础设施。企业级 Agent 的门槛不在于模型能力，而在于工程化——如何让 Agent 融入已有的开发流程而不制造混乱。
