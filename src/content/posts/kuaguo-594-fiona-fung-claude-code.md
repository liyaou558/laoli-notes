> 编程已经不再是瓶颈。它把每个人能做到的事情上限都抬高了。

## 一、核心观点

**1. 编程不再是瓶颈，验证、质量和影响成为新核心。** Anthropic 工程团队代码交付量增长了八倍，但这不意味着需求只是"更快写代码"。当 Claude Code 接管了大量编码工作，真正稀缺的能力变成：验证代码是否真正解决问题、保证质量、理解用户体验是否变好。Fiona 用"bad 与 sad"框架区分不可恢复的致命错误和可恢复的体验痛点——前者必须杜绝，后者累积多了也会致命。

**2. 团队里每个人都变成"构建者"。** 在 Claude Code 团队，不只是工程师提交代码，PM、设计师、数据科学家都开始参与构建。角色边界正在模糊，团队围绕问题而非职能组织。工程师需要更有产品感，PM 也需要更理解工程。"往'构建者'这个方向转——每个人都开始变成构建者。"

**3. 把"什么是好"写进 repo。** 面对 AI 生成代码带来的审查压力，Fiona 的经验是不要只依赖人类审查者。把 spec、设计原则、质量标准等"什么算好"的框架写进代码库，让 Claude Code 基于清晰标准做自动化代码审查。TDD（测试驱动开发）也因此回归——AI 让写测试不再像"交税"。

**4. 不要把动作误当成进展。** 当团队大量使用 AI，人们容易盯着 token 用量、代码行数、PR 数量。但 Fiona 提醒：这些只是"动作"，不等于真正的业务进展。真正该衡量的是：想解决什么问题？产出有没有推动结果？她分享了在 Facebook Marketplace 的教训——环境变化时，指标体系也要更新。

**5. 高能动性必须搭配高责任感。** AI 工具让个人能做的事变多，最有优势的人往往更主动。但团队不能只鼓励自由行动，还必须明确：你要解决什么问题？假设是什么？如何验证？自由必须配套责任。

**6. 面对 AI 焦虑：靠近恐惧。** 对于害怕被 AI 替代的人，Fiona 的建议不是逃避，而是靠近恐惧，问自己"什么还在我控制范围内"。她用年轻时为了支付工程学院学费去做银行柜员的经历说明：行动本身就能对抗不确定性。成长型思维、好奇心和持续学习是适应 AI 时代的核心特质。

**7. AI 不该只属于技术圈。** Fiona 最关心的不是大企业如何用 AI，而是小企业主和普通人能否真正受益。她分享了朋友用 Cowork 分析餐厅菜单和本地定价的例子，鼓励已经深度使用 AI 的人主动帮助身边还没接触的人，从一个具体问题开始打开对话。

**8. Dogfooding 是领导者的基本功。** 无论是在 Marketplace 亲自交易发现骗局，还是在智利实地体验发现网络环境才是增长阻碍——Fiona 反复强调，领导者必须"生活在产品里、呼吸着产品"。数据和仪表盘很重要，但真实使用中的一次小挫败可能揭示数据看不到的问题。

**9. 文化是活的，不是墙上的海报。** 在高速增长的 Anthropic，最让 Fiona 睡不着的不是某个工程问题，而是如何维护团队文化。文化是团队如何讨论问题、如何支持彼此、如何在冲线前回头看有没有人需要帮助。"管理者要敢说'不顺'——别做着火房间里的狗。"

**10. 从六个月路线图到 JIT 即时规划。** Fiona 曾尝试给团队做六个月路线图，三个月后发现文档已不再有用。现在采用更轻量的 JIT（Just-In-Time）规划：看一个月，每周快速检查优先级是否仍然成立，并持续思考能否进一步自动化。"要一直问自己：现在的任何流程，是否还在发挥它本来的作用？"

## 二、关键问答

**Q: 当 AI 让写代码不再是瓶颈，软件团队该怎么工作？**

A: 核心转变是从"发了什么"到"产生了什么影响"。Fiona 的团队用 Claude Code 连接代码库、Slack 和指标，通过 routines 每天自动总结反馈、发现问题、生成 PR。代码审查也不再纯粹依赖人类——把质量标准写进 repo，让 AI 基于标准验证。TDD 回归，因为 AI 让写测试不再像"交税"。但她也提醒，自动审查应该推进到什么程度仍是未解决问题。

**Q: AI 时代招什么样的人？**

A: 两种类型：产品型构建者（能把产品直觉和工程能力结合起来）和深度系统专家。更重要的是软素质——成长型思维、好奇心、持续学习意愿。Fiona 特别强调"野心变得更重要"：AI 抬高了每个人能做到的上限，有野心的人会走得更远。

**Q: AI 编程会让工程师能力退化吗？**

A: Fiona 认为关键是要理解依赖的下一层。虽然 AI 可以帮你编码，但工程师仍然需要知道代码在做什么、为什么这样设计。她用编织做类比——下针、上针、零和一——理解底层原理才能有效使用工具。

**Q: AI 会让编程变得更孤独吗？**

A: 是的，这是 Fiona 真实的担忧。当每个人都在和自己的智能体工作，工程协作可能变孤独。Claude Code 团队因此开始做"两两编程午餐"和黑客松，让大家重新看见彼此如何使用工具、如何构建产品，重新建立团队连接。工程师的快乐来源也在转移——从代码心流转向产品思考。

**Q: 异步智能体时代，管理者日常工作会怎么变？**

A: 管理者可以让 Claude 先帮你发现问题——从写 prompt 到让 AI 写 prompt。Fiona 的团队使用 routines 让智能体自动总结反馈、追踪进展、标记风险。但这也带来新负担：上下文切换变多。高能动性必须配高责任感——自由越大，验证假设的责任越重。

**Q: AI 工具的 ROI 怎么衡量？**

A: 不要用 token 用量、代码行数或 PR 数量。Fiona 的核心原则：不要把动作误当成进展。先定义"我们要解决什么问题"，再看产出是否推动了结果。衡量框架要随着环境变化而更新——她在 Marketplace 的教训是，旧指标在新环境下可能完全失效。

**Q: Fiona 最关心但还没解决的新问题有哪些？**

A: 四个问题：是否还需要独立的 iOS 和 Android 团队？全自动代码审查应该推进到什么程度？如何验证最终体验真的符合目标？以及下一代工程师怎么培养——软件工程会不会变成学徒制？她还提到，过去因为技术不成熟没跑通的 AI 自动化，现在可能值得重新尝试。
