> 嘉宾：Boris Cherny（Anthropic 技术员工，Claude Code 早期核心参与者）

## 一、核心观点

### 1. 编程是模型与现实世界互动的「培养皿」，而非单纯的生产力工具

Boris Cherny 从一个更深层的视角解释 Claude Code 的存在理由：Anthropic 的核心使命是 AI 安全，而编程恰好是研究模型行为的最佳实验环境。代码有严格的正确性约束——能跑就是能跑，不能就是不能，不存在模糊地带。代码有即时的反馈循环——编译错误、测试失败、CI/CD 流水线在秒级时间内给出结果。代码还有真实世界的后果——一个 API 设计失误会影响成千上万的用户。Cherny 将编程描述为模型与现实世界互动的「培养皿」——在这个可控环境里，研究者可以科学地观察模型如何理解意图、如何处理歧义、如何在多重约束之间做出权衡，然后将这些洞察反哺到 AI 安全与对齐研究中。这意味着 Claude Code 不只是一个商业产品，它同时也是 Anthropic 理解自身模型的前沿实验室，商业价值与安全研究在这里形成了罕见的正向循环。

### 2. 模型跃迁是底层驱动力：从 10% 完成率到卸载 IDE 的阶跃式质变

Cherny 坦承 Claude Code 的起点非常卑微——早期版本只能帮他完成大约 10% 到 20% 的代码工作，远不足以替代手工编码。真正的转折来自底层模型的阶跃式能力提升。从 Sonnet 到 Opus 再到更强版本，每一次代际升级都不是线性的性能改进，而是阶跃式的质变——某些之前完全无法完成的任务，在新模型上突然变得可行。这种跃迁不仅改变了 Claude Code 能做什么，更根本地改变了工程师与工具的交互模式：早期需要精确描述每一步，新模型开始展现出对意图的深度理解和自主规划多步骤任务的能力。Cherny 给出了一个标志性时间节点：「到了十一月，我把我的 IDE 卸载了，因为我已经不用它了。」从那一刻起，他连续几个月没有亲手写过一行代码。工作方式从「自己写代码」演变为「同时运行 5 到 10 个 Claude」，再演变为「编写能自动指挥 Claude 的循环」——每一次都是抽象层级的跃迁，而不仅仅是效率的线性提升。

### 3. 工程生产力指数级增长：代码量数倍增长，新人两天上手

Cherny 用具体数据描述了 AI 编程工具带来的生产力跃迁。每位工程师的代码产出量已经实现数倍增长——这不是 20% 或 50% 的渐进式效率提升，而是倍增。他特别指出，Anthropic 此前公开过的「三倍」数据已经过时，实际数字更高。更引人注目的是新人上手时间的压缩：过去一个新入职的工程师需要数周时间熟悉代码库、理解依赖关系、掌握业务逻辑，现在这个周期被压缩到了两天。关键机制不是 AI 替新人写代码，而是 AI 充当了一个永不疲倦的「代码库导航员」——它可以 24 小时回答任何关于代码结构、模块关联、历史决策的问题，让新人不再需要排队等待资深同事的指导时间。Cherny 强调这种增长具有复利效应：越早采用 AI 编程工具的组织，其累积的生产力优势越大。

### 4. 通才的黄金时代：工程师、PM、设计师、数据科学家的边界正在消融

这是 Cherny 本期最核心的判断：「我觉得现在就是通才的黄金时代。」在 Claude Code 团队内部，传统产品流程已经被彻底打破——不再有「研究做完交给设计、设计做完交给产品、产品写完 PRD 交给工程」的串行接力。每个人都在和用户直接交流、定义产品范围、写代码、做数据分析、搭建 dashboard。Cherny 用「技术员工」（Technical Staff）而非「软件工程师」来定义自己的头衔，这本身就是旧角色边界失效的一个信号。更惊人的是，非传统技术角色——财务人员、幕僚长——也开始发布代码。AI 工具正在把「写代码」从一项需要多年训练的专业技能，变成一种通过自然语言就能调用的通用能力。当门槛被降到这种程度时，角色的融合不是组织架构的选择，而是一种自然涌现的结果。

### 5. Claude Cowork 作为存在性证明：八九天，100% 由 Claude Code 构建

Cherny 用一个具体到近乎挑衅的案例证明了上述判断。Claude Cowork 是为不想打开终端的人准备的 AI 编程界面——目标用户是那些有编程需求但不愿或不会使用命令行的群体。这个产品从概念到完成只用了八九天。更关键的是，Cherny 明确表示它「百分之百是用 Claude Code 做出来的」——没有一行人工编写的代码，从架构设计到功能实现到界面构建，完全由 AI 工具自主完成。这不是一个技术演示，而是一个存在性证明：当 Claude Code 的核心开发者选择完全用自己的工具来构建新产品时，这本身就是对工具能力的最强背书。其隐含信息对创业者和产品团队极其明确：验证一个想法的成本正在趋近于零，没有理由不快速实验。

### 6. 「少配人，多给 token」：用约束逼迫自动化，用自动化形成复利

Cherny 给公司和创始人的组织建议简洁而有力：「给大家尽可能多的 token，让他们去实验。」背后的管理哲学与传统思维形成强烈张力。传统组织倾向于多配人来解决问题，但对每个人的 AI 预算严格控制。Cherny 建议完全反转这个逻辑：如果一个项目看起来需要四个工程师，就只放两个进去，再给他们充足的 AI 资源。更少的人力约束会迫使团队寻找自动化路径，而自动化一旦建立就形成永不衰减的复利——今天自动化的一个流程，明天、后天、每一天都在节省时间。多配人只是线性地增加产出，不会产生同样的复利效应。预算结构也需要相应调整：从人力预算向 token 预算倾斜，前期成本可能上升，但长期运行成本持续下降。

### 7. 品味会被追上，价值观不会——人类最后要教给模型的东西

Cherny 在本期收尾处触及了一个很多人在 AI 时代隐隐担忧的终极问题：如果 AI 在所有维度上都超越人类，我们还剩什么？他的回答冷静而坦诚。产品品味——知道什么是好的设计、好的交互、好的用户体验——本质上是对大量样本的模式识别，而这正是 AI 的强项。品味今天仍是人类的优势，但长期也会被模型逐步侵蚀。真正不可替代的是价值观——品味回答「好不好」的问题，价值观回答「该不该」的问题。一个模型可以学会识别什么是好的按钮设计，但它无法从数据中推导出我们应不应该用这个按钮来操纵用户行为。Cherny 用了一个动人的类比收束全篇：「就像我们教孩子怎样做一个好人一样，我们也会教模型怎样做一个好模型。」这不是一个让人安心舒适的答案，但它的诚实本身就有力量——它同时承认了模型的巨大潜力和人类在价值观传递上的终极责任。

## 二、关键问答

**Q: Claude Code 的起源和 Anthropic 的 AI 安全使命之间有什么具体连接？**

A: Cherny 解释，Anthropic 的核心使命是研究和确保 AI 安全，而编程恰好提供了一个独特的实验环境来观察模型行为。代码有严格的正确性约束、即时的反馈循环和真实世界的后果——这种「严格的、即时的、有后果的」互动环境，让研究者能够在受控条件下科学观察模型如何理解意图、处理歧义和做出权衡。编程不是在为模型搭建玩具环境，而是在搭建一个模型行为可以被系统研究的前沿实验室。同时，编程工具的商业收入可以支撑 Anthropic 的安全研究——商业与安全在这里形成了正向循环，而非零和博弈。

**Q: 从只能完成 10%-20% 代码到卸载 IDE，中间发生了什么关键变化？**

A: Cherny 描述的是一个模型能力跃迁与工具链迭代相互加速的过程。底层模型从 Sonnet 到 Opus 再到更强版本，每一次升级都是阶跃式的质变而非线性提升——模型开始展现出对更大上下文的理解能力、跨文件推理能力和自主规划多步骤任务的能力。工具侧也在同步进化——更好的代码库索引、更精准的上下文管理、更可靠的操作执行。当两个方向的能力叠加到临界点时，Cherny 发现自己手动编码的频率越来越低，直到完全卸载 IDE。他的工作方式经历了三个阶段：自己写代码 → 同时运行 5 到 10 个 Claude → 编写自动指挥 Claude 的循环。每一次都是抽象层级的跃迁。

**Q: Anthropic 内部使用 Claude Code 的真实数据是怎样的？**

A: Cherny 给出了几个关键数据点。每位工程师的代码产出量已经实现数倍增长，而且之前公开过的「三倍」数据已经过时——实际数字更高。新人上手时间从几周被压缩到两天。Anthropic 内部每天都在使用 Claude Code 进行开发，这种内部使用形成了产品与模型之间的反馈闭环——工程师在使用中发现问题，反馈直接进入模型和工具的改进循环。Cherny 特别指出这些数据还在快速变化中，任何今天公布的数字都可能很快过时。

**Q: 新人上手从几周到两天的压缩是怎么实现的？**

A: 传统新人上手过程中最大的时间消耗不是写代码本身，而是理解——理解代码库的结构、业务逻辑、团队规范。过去这个过程依赖资深同事充当「导游」，但一个人的时间是有限的。AI 工具可以 24 小时待命，同时回答新人的任何问题，从任何文件、任何函数出发解释上下文，在新人尝试修改代码时即时给出反馈和风险提示。Cherny 强调，两天上手不是指新人两天就能像资深工程师一样工作，而是指两天内就能开始做出有效的代码贡献——AI 充当了永不疲倦的「代码库导航员」。

**Q: 「技术员工」（Technical Staff）这个头衔有什么特殊含义？**

A: Cherny 解释这是 Anthropic 有意弱化职位标签的做法。传统的「高级软件工程师」「首席架构师」等头衔带有权威暗示，可能干扰对想法本身的判断——人们倾向于相信头衔更高的人的意见，即使那个意见在具体问题上未必更正确。去掉高级头衔是为了减少权威对想法判断的干扰，让讨论回归到想法本身的质量。更深层的含义是：在 AI 快速重塑能力的时代，资历正在失效——一个新人可能比资深工程师更擅长使用 Claude Code，而使用工具的能力在当下比过去的经验积累更重要。

**Q: 「通才的黄金时代」具体意味着什么？为什么是现在？**

A: Cherny 的判断建立在两个趋势的交汇点上。第一，AI 编程工具显著降低了「把想法变成代码」的门槛——当自然语言可以驱动代码生成时，写代码不再需要多年的专业训练。第二，AI 工具同时降低了跨领域学习的成本——PM 可以通过 AI 快速理解技术约束，工程师可以通过 AI 快速生成符合设计规范的界面。两个趋势叠加的结果是：过去需要四个不同专业人士串行协作的工作，现在一个 Builder 用 AI 工具可以并行推进。角色边界不是被组织架构打破的，而是被工具能力消解的。Cherny 观察到财务和幕僚长已经在发布代码——转变已经在发生。

**Q: Claude Cowork 是什么？为什么八九天就能做出来？**

A: Claude Cowork 是一个面向非命令行用户的 AI 编程界面——为那些有编程需求但不想或不会使用终端的人准备。Cherny 解释，Claude Code 最初是 CLI 工具，但有很多人不习惯终端环境，需求信号非常明确。选择快速构建的原因很简单：工具能力已经到了。Claude Cowork 百分之百由 Claude Code 构建——没有一行人工代码，从架构到实现到界面完全由 AI 完成。这既是「吃自己的狗粮」的最彻底实践，也是一个存在性证明：AI 编程工具已经达到了可以独立完成完整产品的临界点。

**Q: 为什么不能把 Claude Code 简单地搬到网页里？**

A: Cherny 解释了产品设计上的关键取舍。CLI 的核心优势在于直接访问文件系统和本地工具——读写文件、执行命令、管理进程，这些都是编程工作的核心操作。浏览器环境天然存在沙箱限制，文件系统访问、本地进程管理都不如 CLI 顺手。计算方式也有差异：CLI 模式下模型可以直接写脚本反复运行和调试，而网页端更偏向实时推理。Cherny 的判断是，不同的交互界面服务于不同的使用场景和用户群体——不是要找一个万能形态，而是要为不同的人提供他们最舒适的入口。

**Q: 「少配人，多给 token」这个建议如何落地？**

A: Cherny 的建议可以分解为三个操作层面。第一，团队规模保持克制——不因为 AI 让每个人更高效就同比例增加人手，用更少的人追求更高的产出。第二，token 预算保持慷慨——不给 AI 使用设置过度严格的配额，让工程师有充分空间进行探索性实验。第三，也是核心机制——利用人的稀缺性逼迫自动化。当团队知道自己人手不够时，自然会被迫寻找自动化路径，而自动化一旦建立就形成永不衰减的复利。Cherny 还提到原则文档和 skills 的重要性：将公司的决策原则文档化，让模型也能使用这些原则来做判断，这样可以进一步放大 AI 的自主决策能力。

**Q: 预算从人力转向 token 意味着什么？**

A: Cherny 指出这是一个结构性的成本模型转变。传统软件开发的成本主要是人力——工程师、设计师、PM 的薪酬。AI 时代的成本结构向 token 倾斜：前期在 AI 资源上的投入可能上升，但随着自动化流程的建立和模型推理成本的持续下降，长期运行成本会持续走低。这不是一个简单的「用 AI 替代人力来省钱」的计算，而是一个「投资自动化基础设施以获取长期复利」的战略选择。Cherny 认为组织需要重新思考预算的分配逻辑，将 token 视为一种战略性资源而非运营成本。

**Q: 产品品味真的会被 AI 追上吗？**

A: Cherny 的回答坦诚得令人不安。他认为产品品味——关于什么是好设计、好交互、好体验的判断——在本质上是对大量样本的模式识别。他自己曾经对函数式编程有强烈的审美执念，但最终接受了模型写的 class 风格代码，承认那只是一种个人偏好而非客观优势。随着模型在更多设计案例、用户反馈、A/B 测试结果上的训练，它在品味判断上的表现将持续逼近甚至超越人类平均水平。品味今天仍是短期优势，但长远看这个优势会逐步被侵蚀。

**Q: 当所有优势都被 AI 侵蚀，人类最后还剩下什么？**

A: Cherny 的答案是价值观。品味回答「好不好」，价值观回答「该不该」。模型可以学会识别什么是好的按钮设计，但它无法从数据中自动推导出「我们应不应该用这个按钮来操纵用户行为」的答案。价值观不是训练出来的，是被传递的——就像孩子不是通过规则手册学会做好人的，而是通过观察、反馈和反复的互动内化价值观。Cherny 认为 AI 的对齐也应该是这种模式——不是写一套安全规则丢给模型执行，而是在持续互动中让模型内化价值观。人类在这个过程中的角色是「教育者」——既谦逊（承认模型在很多方面会超越我们），又坚定（坚持价值观的传递是人类最后的、不可外包的责任）。

**Q: 「教模型做一个好模型」和「教孩子做一个好人」这个类比有什么深意？**

A: 这个类比同时说了两件事。第一，它承认模型和人类有本质相似之处——都需要被「养育」而不只是被「训练」。Cherny 暗示 AI 对齐也应该是通过持续的互动和反馈来内化价值观，而非通过冷冰冰的规则手册。第二，它同时确认了人类在这个过程中的不可替代性。就像孩子不能自己教自己做好人——需要父母、老师和社区的引导——模型也不能自己推导出什么是好的价值观。这个类比的力量还在于它的基调：Cherny 选择的动词是「教」，不是「控制」「限制」「对齐」——这些词都有对抗性意味。「教」的基调是合作和成长，而非对抗和约束。

## 三、备忘

- 本期克隆自 Acquired Unplugged（WorkOS 呈现）对 Boris Cherny 的专访。Acquired 是科技商业领域的标杆播客，Unplugged 是其子系列，专注于科技公司的内部故事和产品哲学。30 分钟的紧凑对话信息密度极高，Cherny 几乎每一句话都包含一个可独立展开的深层判断。

- 「卸载 IDE」是本期最具传播力的个人叙事，但它不应该被简化为「AI 替代了程序员」的标题党。Cherny 自己就是 Claude Code 的构建者——他从手工编码转移到用自己构建的工具编码，这更像是工具制造者率先吃狗粮的故事。他的工作方式经历了三个阶段：自己写代码 → 同时指挥多个 Claude → 编写自动指挥 Claude 的循环。真正值得注意的是第三阶段——它意味着抽象层级本身正在成为新的编程对象。

- 「培养皿」的类比值得所有 AI 工具团队深思。如果你把编程工具仅仅看作生产力工具，你会在功能列表上竞争。如果你把它看作观察和研究模型行为的实验环境，你打开的是一扇通向 AI 安全和模型理解的大门。Anthropic 在编程工具上的投入似乎是后一种思维——它服务的不只是用户，还有研究者自己。商业与安全在这里不是零和关系。

- 多入口产品策略（CLI / 桌面 / 移动 / Slack / GitHub）背后的核心洞察：同一个底层 AI 能力，对不同用户群体的价值感知和使用门槛完全不同。CLI 对工程师是生产力倍增器，对 PM 是壁垒。移动端对 PM 是便利工具，对工程师几乎没用。不是要做更多功能，而是要把同一个能力用不同形态触达不同的人。

- 「新人从几周到两天」的数据如果属实，对组织的人力资源策略有深远影响。传统的招聘-培训-上手周期是整个行业运作的底层节奏之一。如果这个节奏被从几周压缩到两天，组织的用人策略、项目排期、团队构成都需要重新思考。但也要警惕「两天上手」的含金量——能开始做有效贡献和能独立负责复杂系统是两回事。

- 「通才的黄金时代」的判断与 Tony Fadell 在 #574 中提出的 Builder 角色融合高度呼应。两位嘉宾从完全不同的角度（AI 工具构建者 vs 硬件产品创造者）得出了相似的结论：AI 正在消解专业角色的边界，催生一种端到端的 Builder 角色。这个交叉验证值得重视。

- Claude Cowork 的案例——八九天、100% AI 构建——是本期最值得被引用的实证。它同时证明了 AI 编程工具的能力边界（能独立完成完整产品）和时间效率（从概念到交付不到两周）。对创业者的隐含信号极为明确：想法验证的成本正在趋近于零，唯一稀缺的是判断力——而这也正是 Cherny 认为最终会被模型追上的东西。

- 「少配人，多给 token」是本期最具操作性的组织建议。与传统管理思维的张力在于：管理者本能地认为多给 headcount 是解决产能问题的方式，但 Cherny 的框架暗示多给人可能恰恰是阻止组织进化的障碍——因为人多就不需要自动化，不需要自动化就没有复利。预算结构也需要相应调整：token 应被视为战略性资源而非运营成本。

- Cherny 对「品味 vs 价值观」的区分是本期最深层的洞察。他本人从函数式编程的执念到接受模型写 class 的经历，是一个微型寓言——很多我们以为是优势的东西，最后都被证明只是个人执念。这个区分回答了很多人在 AI 时代隐隐担心的终极问题：如果 AI 在所有可量化维度上都超越人类，人还剩下什么？答案不是技能、不是品味，而是价值观——关于什么是重要的、什么是正确的判断。

- 「教模型做一个好模型」这个表述值得被定格。Cherny 选择的动词是「教」而非「控制」「限制」「对齐」——这些词都带有对抗性。「教」的基调是合作和成长，呼应了 Anthropic 作为一家 AI 安全公司的底层心态：不是恐惧模型，而是承担起教育者的责任。在一个经常被末日叙事笼罩的领域，这种基调本身就是一个值得注意的信号。

- 数据科学家主动安装 Node.js 和 API key 来使用 Claude Code 做分析，甚至有人用 Claude Code 监控番茄植株并控制营养供给——这些案例说明 AI 编程工具的需求信号已经远远超出了传统软件工程师群体。Claude Code 正在进入主流人群，而这正是 Claude Cowork 诞生的背景。
