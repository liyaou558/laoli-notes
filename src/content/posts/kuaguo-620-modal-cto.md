## 一、核心观点

**1. Agent 体验和开发者体验本质上是同一个东西——你凭什么让 Agent 去读几百个 YAML？**

Akshat 指出，AI Agent 在基础设施上遇到的困境和人类开发者一模一样：被配置文件淹没，缺乏类型安全，需要理解大量隐式约定。他的灵魂拷问是："你为什么要让一个 Agent 去读几百个 Kubernetes 文件、写那种连类型都没有的 YAML？" Modal 的答案是用代码中的装饰器声明需求，运行时自动完成资源供应。无论是人还是 AI，都应该用最简单的方式表达意图，而不是陷入配置文件的泥潭。这个洞察直接推动了 Modal 的 SDK 团队方向从"开发者体验"转向"Agent 体验"。

> "你为什么要让一个 Agent 去读几百个 Kubernetes 文件、写那种连类型都没有的 YAML？"

**2. AI 工作负载和传统 Web 服务器完全不同——Kubernetes 的慢速扩缩容根本不适合**

Akshat 对 K8s 的批评直指本质：K8s 设计之初就是为常驻型 Web 服务准备的，扩缩容是渐进的，调度器假设请求会在秒级时间内被满足。但 AI 推理和训练完全是另一回事——你需要在几分钟内从 1000 张 GPU 扩到 1500 张，而且必须在同一个区域内完成（因为 GPU 之间的通信对延迟极度敏感）。Modal 从第一天起就为"算力密集型、高度突发"的负载设计，引入了"自供应运行时"的概念：用户用代码声明需求（如 `@stub.function(gpu="A100")`），Modal 在毫秒级别完成容器冷启动、GPU 挂载和网络配置。

> "AI 工作负载不是 Web 服务器，Kubernetes 的慢速扩缩容根本不适合。"

**3. 推测解码的加速是乘法级的，不是优化 kernel 那几个百分点能比的**

Akshat 用"接受长度"（Acceptance Length）这个关键指标解释了推测解码的核心逻辑：用小模型（草稿模型）快速预测 token，大模型批量验证。只要草稿模型产出的 token 被足够多地接受，就能获得数倍加速。Modal 开源的 DFlash 在此基础上做了基于块的推测——不是逐个 token 预测，而是整块输出后让大模型一次性验证，进一步放大了加速效果。Akshat 强调这跟优化 kernel 完全不在一个量级上——后者是百分比级别的提升，前者是乘法级别的提升。

> "推测解码的加速是乘法级的，不是优化 kernel 那几个百分点能比的。"

**4. 沙箱的硬边界不可替代——用 LLM 中介权限让人不安**

当 swyx 提出"AI 操作系统的内核可能就是一个 LLM"时，Akshat 明确表达了怀疑。Modal 早在 2023 年就做了沙箱功能，第一个示例就是让 Agent 在沙箱里自我迭代、写代码再执行。但 Akshat 坚持沙箱层面必须有硬边界——网络隔离、文件系统隔离、资源限制——来防止数据泄露。LLM 只能作为软护栏的补充，绝不能成为安全机制的替代品。他还透露沙箱支持 sidecar 模式（一个 pod 里跑多个容器），这让 Agent 可以同时拥有代码执行环境和数据库访问能力，而不会相互污染。

> "沙箱的硬边界不可替代，用 LLM 中介权限让人不安。"

**5. 刻意不做模型 API——纯爱好者市场粘性低，真正的产品公司需要代码级控制力**

与竞争对手 Replicate 的"一键调用模型"API 路线不同，Modal 刻意不提供模型 API 服务。Akshat 的逻辑很清晰：纯爱好者会追逐最新最热的模型，今天用这个明天换那个，粘性极低。而真正在打造产品的公司——比如 Suno、Runway、Cognition——需要的是代码级的灵活性和控制力，不是黑盒 API。这种"筛选效应"让 Modal 的客户更有差异化，对平台的依赖也更深。Akshat 直言："我们最大用例是弹性推理，但刻意避开大语言模型领域。"这一选择帮助 Modal 避开了模型 API 层的红海竞争。

> "纯爱好者市场粘性低。真正在打造产品的公司需要的是代码级的灵活性，而不是一个黑盒 API。"

**6. 在 17 家云厂商之上构建可靠性层——让用户接触到的 GPU 容量比自己能拿到的多得多**

Modal 没有自己的数据中心，而是在全球 17 家云厂商和裸金属提供商之上构建了一个"超级云"。Akshat 透露 Modal 内部有一个"计算策略团队"（Compute Strategy Team），专门做跨供应商的容量规划和价格优化。他们在这 17 家之上加了一层可靠性抽象，让 GPU 掉线、节点故障或区域容量不足对用户工作负载完全透明。结果是：单个用户靠自己可能只能接触到几百张 GPU，但通过 Modal 能触及的容量池远超这个数。批量处理层更是利用这个优势——只要用户不在乎延迟，就能拿到便宜得多的价格。

> "在 17 家云厂商之上构建可靠性层，让用户接触到的容量比自己能拿到的多得多。"

**7. 基础设施最激动人心的时代——之前其实无聊了好一阵子**

swyx 回顾了过去几年的变化："你很难让人对数据基础设施兴奋起来，但现在大家都在意了。"Akshat 完全认同，并指出规模是一切变化的底层驱动力：AI 需要的算力规模让基础设施重新变成了一个性感且充满挑战的领域。传统云架构的假设被逐一打破——网络拓扑、存储带宽、电源密度、散热方案——每一个环节在大规模 GPU 集群面前都需要重新设计。对基础设施工程师来说，这是十年来最好的时代。

> "基础设施最激动人心的时代，之前其实无聊了好一阵子。"

## 二、关键问答

**Q1：Modal 到底是一个什么平台？**

Akshat 将 Modal 定义为一个从零为 AI 应用构建的云平台，覆盖弹性推理、分布式训练、批处理和 Agent 沙箱四大场景。与传统云平台不同，Modal 的核心理念是"代码即配置"——用户用 Python 装饰器声明需求（GPU 类型、内存大小、超时时间等），运行时自动完成供应，无需写 YAML 或管理 K8s 集群。

**Q2：为什么传统云架构（尤其是 Kubernetes）从根本上不适合 AI 工作负载？**

三个核心矛盾：第一，K8s 的扩缩容是渐进的（分钟级），而 AI 负载是突发性的（需要秒级从几百张 GPU 扩到上千张）。第二，GPU 之间的通信对延迟极度敏感，要求调度在同一区域内完成，传统云调度器不感知物理拓扑。第三，AI 任务的生命周期差异巨大——推理请求可能只需几百毫秒，训练任务可能跑几周——K8s 的 pod 抽象无法优雅地统一处理。

**Q3：DFlash 推测解码具体是怎么工作的？为什么是"乘法级"加速？**

DFlash 的核心思路是基于块的推测解码：用一个轻量级草稿模型快速生成一整块 token（而非逐个），然后让大模型一次性验证这一整块。关键指标是"接受长度"——被大模型接受的 token 越多，有效加速比就越高。与优化单个 kernel 的百分比级提升不同，推测解码可以直接将吞吐量翻倍甚至翻几倍，是真正的乘法级加速。

**Q4：Modal 的沙箱功能有什么特别之处？**

Modal 在 2023 年就推出了沙箱功能，第一个 demo 就是让 Agent 在沙箱中自我迭代。沙箱提供硬边界隔离（网络、文件系统、资源限制），同时支持 sidecar 模式——一个 pod 内跑多个容器，让 Agent 可以同时访问代码执行环境和数据库。此外，Modal 还有一个隐藏功能 I6PN：用 IPv6 构建的私有覆盖网络，让沙箱之间可以安全通信。

**Q5：为什么 Modal 刻意不做模型 API？**

Akshat 认为模型 API 是一个粘性极低的市场——纯爱好者会不断追逐最新模型，频繁切换供应商。而 Modal 的目标客户是真正在打造产品的公司（如 Suno、Runway、Cognition），他们需要的是代码级的灵活性和控制力，不是黑盒 API。这种筛选让 Modal 的客户更稳固，也避开了模型 API 层的价格战。

**Q6：在 17 家云厂商之上做容量规划有多难？**

Modal 的"计算策略团队"需要在全球 17 家云厂商和裸金属提供商之间做实时容量规划和价格优化。挑战在于：不同区域的 GPU 可用性波动很大，价格也在实时变化。Modal 在上层构建了可靠性抽象，让 GPU 掉线或故障对用户透明。结果是个体用户能接触到的 GPU 容量远超自己谈判能拿到的量。

**Q7：Agent 时代为什么可观测性比代码本身更重要？**

Akshat 指出，当 Agent 在沙箱中自主执行多步骤任务时，开发者需要知道的不只是"代码有没有 bug"，而是"Agent 到底做了什么、为什么做了这个决定"。传统的日志和监控工具无法满足这个需求。Modal 正在将可观测性作为 SDK 的一等公民来设计，让 Agent 的每一步操作都可追溯。

**Q8：CI/CD 怎么用内存快照让 Agent 驱动的 CI 更高效？**

Akshat 批评传统 CI/CD 的巨大浪费：每次运行都要从头拉代码、安装依赖、构建环境，大量时间花在重复工作上。Modal 的方案是用内存快照——把构建好的环境状态保存为快照，下次 CI 运行时直接从快照恢复，跳过所有重复步骤。对 Agent 驱动的 CI（一个 Agent 反复修改代码并测试）来说，这种秒级恢复的能力尤为关键。

**Q9：多节点训练的方向是什么？**

Modal 的多节点训练聚焦小规模后训练（Post-Training），而不是大规模预训练。Akshat 认为预训练是一个已经被少数大玩家锁定的市场，算力需求动辄上万张 GPU 且周期长。而后训练和微调的需求更碎片化、更灵活，正是 Modal 擅长的高弹性场景。

**Q10：批量处理层怎么拿到便宜得多的价格？**

核心逻辑是利用"延迟套利"：很多计算任务对延迟不敏感（如批量推理、数据预处理），可以在 GPU 需求低谷时运行。Modal 利用 17 家供应商的容量池，找到当前最便宜、有空闲 GPU 的区域，把任务调度过去。用户不在乎具体在哪跑，只要结果按时返回就行——代价是延迟稍高，但价格大幅下降。

**Q11：I6PN 是什么？**

I6PN 是 Modal 的一个隐藏功能，利用 IPv6 为沙箱构建的私有覆盖网络。它让不同沙箱之间可以安全、低延迟地通信，而无需暴露在公网上。对于需要多个沙箱协作的 Agent 场景（如一个沙箱查数据库、另一个执行代码），I6PN 提供了干净的网络隔离和高效的数据通道。

**Q12：Modal 的创业起点是什么？**

Akshat 和团队最初的痛点是：为什么所有工作流编排产品都那么难用？他们意识到问题不在编排本身，而在底层运行时。传统的 serverless 函数平台无法处理 GPU 负载，而 K8s 又过于复杂。Modal 从"构建更好的运行时"出发，从 serverless 函数逐步演进到 GPU 推理，最终覆盖了整个 AI 工作负载谱系。

**Q13：如何看待托管 Agent 平台与基础设施层的关系？**

Akshat 认为两者是互补而非竞争。托管 Agent 平台（如 LangChain 生态）解决的是"Agent 逻辑怎么编排"，而 Modal 解决的是"Agent 跑在哪里、怎么分配算力"。如果基础设施层足够好，Agent 平台可以更专注于上层逻辑。但如果基础设施层不好用，Agent 平台也会被拖累——这也是为什么 Modal 越来越重视 Agent 体验。
