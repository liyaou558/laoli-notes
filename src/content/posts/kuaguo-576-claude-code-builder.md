## 一、核心观点

### 1. Claude Code 起源：从内部原型到卸载 IDE

Boris Cherny 讲述了一个令人震撼的个人编程范式转变。Claude Code 最初只是 Anthropic 内部的一个实验性原型——早期版本的代码完成率只有 10% 到 20%，远不足以替代手工编码。但随着模型能力的跃迁和工具的持续迭代，情况发生了质变。Cherny 给出了一个标志性时间节点：「到了十一月，我把我的 IDE 卸载了，因为我已经不用它了。」从那一刻起，他连续几个月没有亲手写过一行代码——所有编码工作完全通过 Claude Code 完成。这不是一个渐进式的辅助工具升级，而是一次编程范式的断裂：工程师从「写代码的人」变成了「指挥代码生成的人」。

### 2. 编程是模型与现实世界互动的「培养皿」

Cherny 从 Anthropic 的底层使命出发，揭示了 AI 安全与编程工具之间的深层连接。Anthropic 的核心关切是 AI 安全，而编程恰好提供了一个独特的环境：它是模型与现实世界互动的最直接、最可量化的通道。代码有明确的正确性标准（能不能跑、有没有 bug）、有即时的反馈循环（编译错误、测试失败）、有真实世界的后果（产品功能、用户体验）。Cherny 将编程描述为模型与现实世界互动的「培养皿」——在这个可控的环境里，研究者可以观察模型如何理解指令、如何做出决策、如何处理不确定性，进而将这些洞察反馈到 AI 安全和对齐的研究中。编程工具不只是生产力工具，它是理解模型行为的前沿实验室。

### 3. 模型跃迁：从 Sonnet/Opus 到阶跃式能力变化

Cherny 描述了模型能力的非线性跃迁过程。从 Sonnet 到 Opus 再到更强版本，每一次代际升级都不是线性的性能提升，而是阶跃式的质变——某些之前完全无法完成的任务，在新模型上突然变得可行。这种跃迁不仅改变了 Claude Code 能做什么，更改变了工程师与工具的交互模式：早期版本需要工程师精确描述每一个步骤，新版本开始展现出对意图的理解和自主规划能力。与此同时，Anthropic 采取了多入口产品策略——CLI、桌面应用、移动端、Slack 集成、GitHub 集成——让不同类型的用户都能以自己最舒适的方式接入 AI 编程能力。Cherny 的判断是：模型能力的阶跃式增长是底层驱动力，多入口策略是让这种能力触达更多人的分发管道。

### 4. 工程生产力指数增长：从几周上手到两天

Cherny 用具体数据描述了 AI 编程工具带来的生产力跃迁。每位工程师的代码产出量实现了数倍增长——这不是 20% 或 50% 的效率提升，而是倍增。更引人注目的是新人上手时间的变化：过去一个新入职的工程师需要几周时间才能熟悉代码库并开始有效贡献，现在这个周期被压缩到了两天。这种压缩不仅是代码生成速度的问题，更深层的原因是 AI 工具能帮助新人快速理解代码库的结构、依赖关系和业务逻辑——它充当了一个永不疲倦的代码库导航员。Cherny 强调，这种生产力增长不是线性的，而是具有复利效应：越早采用 AI 编程工具的组织，其累积的生产力优势越大。

### 5. 通才的黄金时代：工程师/PM/设计师/数据科学家边界正在合并

这是 Cherny 本期最具洞察力的判断之一：「我觉得现在就是通才的黄金时代。」AI 编程工具正在消解传统的角色边界。过去，工程师写代码、PM 写需求文档、设计师画原型、数据科学家跑模型——这些角色之间的工作流是串行的、有明确交接的。AI 工具改变了这一切：PM 可以直接用 Claude Code 搭建可交互的原型而不仅是写 PRD，设计师可以把视觉想法转化成可运行的界面而不仅是 Figma 文件，数据科学家可以独立完成从数据处理到应用部署的全链路而不仅是输出分析报告。甚至更令人惊讶的是——Cherny 观察到，财务人员、幕僚长这类传统意义上的「非技术角色」也开始发布代码。所有这些角色的共同新名字是 Builder——一个正在涌现的通才型角色，其核心能力不再是某个狭窄领域的深度技能，而是「用 AI 工具把想法变成产品」的端到端能力。

### 6. Claude Cowork：八九天、百分百用 Claude Code 构建

Cherny 用一个具体案例证明了上述通才论点的可行性。Claude Cowork 是一个为不想打开终端的人准备的 AI 编程界面——它的目标用户是那些有编程需求但不愿或不会使用命令行的群体。这个产品从概念到完成只用了八九天。更关键的是，Cherny 明确表示：「它百分之百是用 Claude Code 做出来的。」没有人工写任何一行代码，从架构设计到功能实现到界面构建，完全由 AI 工具自主完成。这个案例的力量不在于技术上的炫技，而在于它证明了 AI 编程工具已经达到了一个临界点：一个 Builder（即使这个人本身就是 Claude Code 的核心开发者）可以用纯自然语言对话，在极短的时间内完成一个完整产品的构建。

### 7. 组织建议：「少配人，多给 token」

Cherny 给出的组织建议简洁而有力——「给大家尽可能多的 token，让他们去实验。」背后是一套更深层的管理哲学：用资源约束逼迫自动化，用自动化形成复利。传统组织倾向于「多配人，少给资源」——增加 headcount 来解决问题，但对每个人的 AI 工具预算严格控制。Cherny 建议完全反转这个逻辑：少配人——保持团队精简，制造适度的资源约束，迫使团队寻找自动化路径；多给 token——给予每个人充足的 AI 使用额度，鼓励大量实验和探索。这种机制的核心洞察是：自动化一旦建立就形成复利——今天自动化的一个流程，明天、后天、每一天都在节省时间。而多配人只是线性地增加产出，不会产生同样的复利效应。

### 8. 品味、价值观与人类最后要教给模型的东西

Cherny 在本期最后触及了一个深层问题：当 AI 在代码生成速度、质量甚至产品品味上都逐渐追上甚至超越人类时，人类还剩下什么不可替代的东西？他的回答是价值观。产品品味——知道什么是好的设计、好的交互、好的用户体验——这些东西也会被模型追上，因为在某种意义上品味是大量样本训练后的模式识别，而这正是 AI 擅长的事。但价值观不同——什么是重要的、什么是对的、什么是不该做的——这些不是数据训练能自动得出的结论。Cherny 用了一个动人的类比：「就像我们教孩子怎样做一个好人一样，我们也会教模型怎样做一个好模型。」人类最后要教给模型的不是技能，而是价值观。这既是 Anthropic 使命的终极表达，也是对 AI 时代人类角色的重新定义。

## 二、关键问答

**Q: Claude Code 从只能完成 10-20% 代码到让人卸载 IDE，中间发生了什么？**

A: Cherny 描述的过程并非一夜之间的突变，而是一个模型能力跃迁与工具链迭代相互加速的过程。早期 Claude Code 只能处理简单、局部的代码补全——补一行、改一个函数。但随着 Anthropic 底层模型从 Sonnet 到 Opus 再到更强版本的阶跃式升级，模型开始展现出对更大范围上下文的理解能力、跨文件的推理能力、以及自主规划多步骤编码任务的能力。工具侧也在同步进化——更好的代码库索引、更精准的上下文管理、更可靠的操作执行。当这两个方向的能力叠加到某个临界点时，Cherny 发现自己在 IDE 中手动编码的频率越来越低，直到有一天他意识到已经几个月没有打开 IDE 了。这不是一个「AI 替代了编码」的简单故事，而是一个「编码的定义本身发生了变化」的深层转变。

**Q: 「编程是模型和现实世界互动的一种方式」——这句话的深层含义是什么？**

A: 这句话揭示了 AI 安全研究与编程工具之间的战略性连接。AI 安全研究者面临一个根本困难：如何在模型变得更强大之前，理解和预测它在真实世界中的行为？编程提供了一个近乎完美的实验环境。代码具有严格的正确性约束——要么能跑要么不能跑，不存在模糊地带。代码有即时的反馈循环——编译器、测试框架、CI/CD 流水线会在秒级时间内告诉你结果。代码有真实世界的后果——一个 API 设计失误会影响成千上万的用户。这种「严格的、即时的、有后果的」互动环境，让研究者能够在受控条件下观察模型如何理解意图、如何处理歧义、如何在多重约束之间做出权衡。Cherny 的「培养皿」比喻精准地捕捉了这一点：编程不是在为模型搭建玩具环境，而是在搭建一个模型行为可以被科学观察的前沿实验室。

**Q: 模型的多入口产品策略（CLI/桌面/移动/Slack/GitHub）背后是什么逻辑？**

A: Cherny 揭示的逻辑是：不同用户群体对 AI 编程工具的使用场景和心理门槛完全不同。CLI 服务于硬核工程师——他们在终端里工作，追求速度和可控性。桌面应用服务于需要可视化界面的开发者——看图、看 diff、管理多个项目。移动端服务于需要在手机上快速查看或审批代码的场景。Slack 集成把 AI 编程能力嵌入团队日常沟通——在讨论中直接生成、修改、审查代码。GitHub 集成则把 AI 嵌入已有的代码审查和 CI/CD 工作流。这不是功能冗余，而是能力分发——同一个底层模型能力，通过不同的交互界面触达不同场景、不同角色、不同心理门槛的用户。Cherny 的隐含判断是：AI 编程工具的普及瓶颈不在模型能力，而在交互门槛——多入口策略本质上是在降低这个门槛。

**Q: 新人上手从几周到两天，这种压缩是怎么实现的？**

A: 传统新人上手过程最大的时间消耗不是写代码本身，而是理解——理解代码库的结构（这个项目有哪些模块？它们之间怎么关联？）、理解业务逻辑（这个函数为什么存在？它服务于什么业务场景？）、理解团队的编码规范和约定俗成的实践。一个资深同事充当「导游」时，他的时间是有限的——每天能回答的问题、能做的代码走查都有上限。AI 工具在这方面有天然优势：它可以 24 小时待命，可以同时回答新人的任何问题，可以从任何文件、任何函数出发解释上下文，可以在新人尝试修改代码时即时给出反馈和风险提示。Cherny 描述的两天上手不是指新人两天就能像资深工程师一样工作，而是指两天内就能开始做出有效的代码贡献——AI 充当了那个永不疲倦的「代码库导航员」。

**Q: 「通才的黄金时代」具体意味着什么？为什么现在发生？**

A: Cherny 的判断建立在两个趋势的交汇点上。第一个趋势是 AI 编程工具显著降低了「把想法变成代码」的门槛——当自然语言可以驱动代码生成时，写代码不再需要多年的专业训练。第二个趋势是 AI 工具同时降低了跨领域学习的成本——一个 PM 可以通过与 AI 对话快速理解技术约束，一个工程师可以通过 AI 快速生成符合设计规范的前端界面。这两个趋势叠加的结果是：过去需要四个不同专业人士串行协作才能完成的工作，现在一个 Builder 用 AI 工具可以并行推进。角色边界不是被组织架构打破了，而是被工具能力消解了——当一个角色有能力独立完成端到端的工作时，组织就没有理由继续维持原有的分工结构。这不是一个遥远的未来——Cherny 观察到财务和幕僚长已经在发布代码，这说明转变已经在发生。

**Q: Claude Cowork 百分百用 Claude Code 构建这件事为什么重要？**

A: 这不是一个技术演示，而是一个存在性证明。当 Claude Code 的核心开发者选择完全用自己的工具来构建一个新产品时，这本身就是对工具能力的最强背书——吃自己的狗粮，而且吃得很彻底。更重要的含义是：如果连一个复杂的、面向非技术用户的 AI 编程界面都可以在八九天内完全由 AI 构建，那么大量更简单的内部工具、原型产品、一次性脚本的生产成本正在趋近于零。Cherny 展示的不是「AI 能写代码」，而是「AI 能独立完成产品」——从理解需求到架构设计到实现到交付的全流程。这对创业者和产品团队的隐含信息是：验证一个想法的成本已经降到极低，没有理由不快速实验。

**Q: 「少配人，多给 token」的组织建议如何落地？有没有风险？**

A: Cherny 的建议可以分解为三个操作层面。第一，在团队规模上保持克制——不因为 AI 让每个人更高效就同比例增加人手，而是用更少的人追求更高的产出。第二，在 token 预算上保持慷慨——不要给 AI 使用设置过度严格的配额，让工程师有充分的空间进行探索性实验。第三，也是最重要的——利用人的稀缺性来逼迫自动化。当一个团队知道自己没有足够的人手来完成所有工作时，自然会被迫寻找自动化路径，而一旦自动化路径被找到并验证，它就形成了永不衰减的复利。风险主要在于两个方向：一是过度自动化可能导致团队失去对系统底层理解的能力，当自动化出了问题没人知道怎么修；二是 token 预算如果完全不加管理，可能导致大量低效甚至无效的 AI 调用。Cherny 的框架应该被理解为「给定目前的预算水平，向 token 侧倾斜」，而非「无限制地给所有人无限 token」。

**Q: 产品品味真的会被 AI 追上吗？人类最后剩下什么？**

A: Cherny 的判断冷静而坦诚。他认为产品品味——关于什么是好设计、好交互、好体验的判断——本质上是对大量样本的模式识别，而这恰恰是 AI 的强项。随着模型在更多设计案例、用户反馈、A/B 测试结果上的训练，它在品味判断上的表现将持续逼近甚至超越人类平均水平。但 Cherny 划出了一条关键边界：价值观不属于这个范畴。品味回答的是「好不好」的问题，价值观回答的是「该不该」的问题。一个模型可以学会识别什么是好的按钮设计，但它无法从数据中推导出「我们应不应该用这个按钮来操纵用户行为」的答案。这就是为什么 Cherny 说「就像我们教孩子怎样做一个好人一样，我们也会教模型怎样做一个好模型」——价值观不是训练出来的，是被传递的。人类最后要教给模型的不是技能，而是判断力——而且是关于对错的判断力，不只是关于优劣的判断力。

**Q: 「教模型做一个好模型」和「教孩子做好人」这个类比有什么深意？**

A: 这个类比的力量在于它同时说了两件事。第一，它承认模型和人类有本质相似之处——它们都需要被「养育」而不只是被「训练」。孩子不是通过规则手册学会做好人的，而是通过观察、反馈、反复的互动内化价值观。Cherny 暗示 AI 的对齐也应该是这种模式——不是写一套冷冰冰的安全规则丢给模型执行，而是在持续的互动中让模型内化价值观。第二，它同时承认人类在这个过程中的不可替代性。就像孩子不能自己教自己做好人——需要父母、老师、社区的引导——模型也不能自己推导出什么是好的价值观。人类在这个类比中承担的是「教育者」的角色，这一定位既谦逊（承认模型在很多方面会超越我们）又坚定（坚持价值观的传递是人类最后的、不可外包的责任）。

## 三、备忘

- 本期克隆自 Acquired Unplugged（WorkOS 呈现）对 Boris Cherny 的专访。Acquired Unplugged 是知名商业播客 Acquired 的子系列，专注于科技公司的内部故事和产品哲学。本期是 30 分钟的紧凑对话，信息密度极高——Cherny 几乎每一句话都包含一个可展开的深层判断。

- 「卸载 IDE」的故事是本期最具传播力的个人叙事，但它不应该被简化为「AI 替代了程序员」。Cherny 自己就是 Claude Code 的构建者——他从手工编码转移到用自己构建的工具编码，这更像是一个工具制造者率先吃狗粮的故事，而非一个被动替代的故事。

- 「培养皿」的类比值得所有 AI 工具团队深思。如果你把编程工具仅仅看作生产力工具，你会在功能列表上竞争。如果你把它看作观察和研究模型行为的实验环境，你打开的是一扇通向 AI 安全和模型理解的大门。Anthropic 在编程工具上的投入似乎是后一种思维——它服务的不只是用户，还有研究者自己。

- 多入口产品策略在 Claude Code 上的实践，对任何做 AI 产品的团队都有参考价值。核心洞察：同一个底层 AI 能力，对不同用户的价值感知和使用门槛是完全不同的。CLI 对工程师是生产力倍增器，对 PM 是壁垒。移动端对 PM 是便利工具，对工程师几乎没用。不是要做更多功能，而是要把同一个能力用不同形态触达不同的人。

- 「新人从几周到两天」的数据如果属实，对组织的人力资源策略有深远影响。传统的招聘-培训-上手周期是整个行业运作的底层节奏之一。如果这个节奏被从几周压缩到两天，组织的用人策略、项目排期、团队构成都需要重新思考。但也要警惕「两天上手」的含金量——能开始做有效贡献和能独立负责复杂系统是两回事，Cherny 指的是前者。

- 「通才的黄金时代」的判断与 Tony Fadell 在 #574 中提出的 Builder 角色融合高度呼应。两位嘉宾从完全不同的角度（一个是 AI 工具构建者，一个是硬件产品创造者）得出了相似的结论：AI 正在消解专业角色的边界，催生一种端到端的 Builder 角色。这个判断的交叉验证值得重视。

- Claude Cowork 的案例——八九天、100% AI 构建——是本期最值得被引用的实证。它同时证明了 AI 编程工具的能力边界（能独立完成完整产品）和时间效率（从概念到交付不到两周）。对创业者的隐含信号很明确：想法验证的成本正在趋近于零，唯一稀缺的是判断力和品味——而这也正是 Cherny 认为最终也会被模型追上的东西。

- 「少配人，多给 token」是本期最具操作性的组织建议。它与传统管理思维的张力在于：管理者本能地认为多给 headcount 是解决产能问题的方式，但 Cherny 的框架暗示多给人可能恰恰是阻止组织进化的障碍——因为人多就不需要自动化，不需要自动化就没有复利。

- Cherny 对「品味 vs 价值观」的区分是本期最深层的洞察。它回答了一个很多人在 AI 时代隐隐担心的终极问题：如果 AI 在所有可量化的维度上都超越人类，人类还剩下什么？答案不是技能（会被追上）、不是品味（也会被追上）、而是价值观——关于什么是重要的、什么是正确的判断。这不是一个让人安心舒适的答案，但它的诚实本身就有力量。

- 「教模型做好模型」这个表述值得被定格。它不是「控制模型」「限制模型」「对齐模型」——这些词都有一种对抗性的意味。Cherny 选择的动词是「教」，就像教孩子一样。这个措辞的基调是合作而非对抗，是成长而非约束。在一个经常被末日叙事笼罩的领域，这种基调本身就是一个值得注意的信号——它来自 Anthropic 内部最接近模型和工具的人，或许反映了这个组织真实的底层心态。
