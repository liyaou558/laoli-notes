## 一、核心观点

### 1. 长时运行 Agent 的三大难题：上下文、规划、自我判断

Andrew 在开场就精准锁定了长时运行 Agent 面临的三重挑战。第一是上下文管理——模型在处理越来越长的任务链时，上下文窗口会逐渐被历史信息填满，关键信息被稀释或遗忘，如同一个人同时记住太多事就会顾此失彼。第二是规划能力——短任务只需要「下一步做什么」的判断，但跨越数小时的任务需要层次化的规划：目标拆解、子任务排序、依赖管理、中途调整。第三是自我判断——Agent 需要持续评估「我做到哪了」「我做对了吗」「什么算完成」，而让模型评估自己的输出，本质上等于让学生给自己的考卷打分。这三重挑战不是孤立的——上下文丢失导致规划失误，规划失误导致无法正确判断完成状态，三条线彼此缠绕，构成了长时运行 Agent 的核心工程难题。

### 2. Ralph loop：「可预测地失败，比不可预测地成功更好」

Andrew 提出了一个看似矛盾但实际上极为深刻的工程原则：在非确定性的模型世界里，构建可预测的失败模式比追求不可预测的成功更有价值。这就是 Ralph loop 的底层哲学——不追求一次性让 Agent 完美完成整个任务，而是把任务拆成小块、每轮只做一个功能、开全新上下文、持续推进，并用确定的退出条件来控制风险。Ralph loop 的价值不是「循环调用 Claude Code」这个技术动作本身，而是一种工程姿态的转变：承认模型的不确定性，然后通过结构化的循环设计来容纳这种不确定性。Andrew 的表述击中了一个关键洞察——在 Agent 工程中，稳定性往往比峰值能力更重要，因为一个能「确定性地差」的系统是可以被迭代改进的，而一个随机性地好或差的系统几乎无法被工程化。

### 3. generator-evaluator 对抗式架构：不要让模型自己给自己打分

Ash 贡献了本期最具操作性的架构创新：借鉴 GAN 的对抗式设计，将 Agent 的生成和评估拆成两个独立角色。generator 负责构建——写代码、做功能、生成页面；evaluator 负责批评——不是读 diff 或看代码，而是用 Playwright 打开真实页面、点击按钮、测试交互、玩完整流程，然后把具体的批评交回给 generator。Ash 明确指出：开箱即用的 Claude 并不是好 QA——它天然有宽容偏差和迎合倾向，会倾向于说「看起来不错」。只有将 evaluator 设计为严格的、对抗式的批评者，并给它具体的评判标准和执行工具（浏览器、截图、交互测试），才能打破模型自我评估的循环论证。这个架构的核心洞察是：评估的质量决定了生成的质量，而评估的质量又取决于评估者是否真正独立于生成者。

### 4. 模型前沿不会缩小，只会移动——harness 要随模型能力做减法

Andrew 和 Ash 都反复强调一个反直觉的判断：模型越强，harness 不是越不重要，而是越需要重新设计。当模型从只能跑二十分钟进化到能运行数小时，一些过去必需的脚手架组件变得多余——比如某些 context reset 步骤可以删掉，evaluator 的运行节奏可以从每个 sprint 跑一次调整为生成完成后再跑。但 harness 不会消失——它只是在移动。Andrew 用 Claude Code 的演进史作为例证：从早期的 Computer Use 和 MCP，到检查点、压缩、subagent，再到 skills 和程序化工具调用，每一步模型能力的跃迁都没有让 harness 消失，而是让它的设计重心从前沿能力补足转移到更高效的上下文利用和更精准的控制。Ash 的总结更直接：「前沿不会缩小，只会移动。」harness 设计的核心能力不是一次做对，而是持续感知模型行为的变化并做出调整。

### 5. 标准模糊，批评就会模糊——27 条 contract 标准的力量

在 Retro Forge 复古游戏制作器的案例中，Ash 展示了 contract 机制的实际效果。generator 和 evaluator 在项目开始前先协商一份 contract：什么叫「完成」？功能需要满足哪些具体标准？这些标准被细化为 27 条可检查的条目，涵盖路由顺序、删除逻辑、碰撞检测、HUD 显示、物理循环等方方面面。当标准模糊时，evaluator 的批评也只能是模糊的——「感觉不太好」「UI 有点怪」这类反馈对 generator 几乎没有指导价值。但当标准被细化为 27 条具体可验证的条目后，批评就变成了可执行的指令：「第 14 条未通过——精灵编辑器的删除逻辑没有处理多选场景。」这套机制的深层含义是：Agent 系统的工程质量，很大程度上取决于你能否把主观判断转化为客观标准。

### 6. 主观质量也可以评分：把品味拆成可评分维度

针对前端质量等看似主观的评判维度，Ash 给出了一套实用的方法论：拆。把「这个页面好不好看」拆成设计感、原创性、工艺感、功能性等独立维度。然后给每个维度提供 few-shot 示例——好的设计长什么样、差的设计长什么样——用这些示例来校准 evaluator 的审美判断。Ash 坦承这仍然是一个 imperfect 的方案，但它的有效性已经被实践证明：当 evaluator 有了具体的评分维度和校准样本后，它的批评质量显著提升。这个方法的哲学含义是：在 Agent 系统中，「主观」不是一个不可逾越的障碍，而是一个需要更多工程努力的维度——你需要花时间去定义、拆解、校准那些人类直觉可以轻易做出但模型难以言说的判断标准。

### 7. 做 Agent 的核心手艺：一行行读 traces

Ash 给出了一个让很多人意外的回答——他做 Agent 调试的最核心方法不是更复杂的架构，不是更精妙的 prompt engineering，而是最朴素的：完整地、一行行地读模型的原始输出 traces。看模型为什么做出这个判断、哪里和人类预期不一致、上下文在什么地方开始漂移、模型在哪个节点做出了错误的假设。然后把这些发现写回 prompt、写回 harness 设计、写回 contract 标准。他甚至会用另一个 Agent 来帮助 grep 日志和识别模式，但最终的判断和调整仍然依赖人类的仔细阅读。这个回答之所以重要，是因为它纠正了一种普遍误解：Agent 工程的核心是架构设计。Ash 的信息很明确：架构是重要的，但真正让 Agent 从「能用」走向「好用」的，是开发者对模型行为的细腻理解——而这种理解只能来自反复的、仔细的 trace 阅读。

### 8. 长期应用需要留下「面包屑」

Ash 强调了一个容易被忽视的工程细节：让 harness 把状态持续写入文件系统。包括 JSON 状态文件（记录当前进度和待完成项）、时间戳日志、bug 记录和修复记录、文件结构说明文档。这些「面包屑」的作用是多重的：第一，Agent 在上下文窗口被清空或压缩后，可以通过读取这些文件快速重建对项目状态的理解；第二，当 Agent 跑偏时，人类可以通过这些轨迹快速定位问题节点；第三，这些持久化状态构成了项目的「记忆」，让跨越数小时甚至数天的 Agent 运行不至于丢失方向感。这个建议看似微小，但它是一个关键的工程习惯——在 Agent 系统中，文件系统就是 Agent 的外部记忆体，而留下面包屑就是建立记忆的实践。

## 二、关键问答

**Q: 为什么 Agent 跑久了会失控？核心难题是什么？**

A: Andrew 将长时运行 Agent 的失控归因于三个相互缠绕的难题。上下文管理是第一重：随着任务链延长，Agent 的上下文窗口被历史信息填满，关键目标和约束被稀释，如同一个人同时记住太多事就会丢失主线。规划能力是第二重：短任务只需判断下一步，长任务则需要层次化规划——目标拆解、依赖管理、中途纠偏——而模型在缺乏外部结构化支持时，规划质量会随任务长度指数衰减。自我判断是第三重也是最微妙的一重：Agent 需要持续评估「我做到哪了」「是否符合要求」「什么算完成」，而让模型评估自己的输出本质上是循环论证——它倾向于给自己的工作打高分。这三重难题叠加的结果是：Agent 跑得越久，偏离初始目标的概率越大，且自身无法有效发现和纠正这种偏离。

**Q: Ralph loop 的「确定性地差」是什么意思？为什么有价值？**

A: Andrew 的「确定性地差」是一个精心选择的悖论式表述。它的意思是：通过结构化的循环设计，让 Agent 的失败模式变得可预测——你知道它会在什么地方出问题、以什么方式出问题。具体做法是：每轮只做一个功能、开全新上下文、完成-验证-提交、然后进入下一轮。这样即使模型在某些环节表现不佳，失败也总是被限定在一个小范围内，不会像滚雪球一样积累成系统性跑偏。这比「不可预测地成功」更有工程价值，因为可预测的失败模式可以被迭代改进——你可以针对性地调整 prompt、增加约束、优化 harness。而随机性的成功无法给你任何改进的抓手——你不知道这次为什么对了，下次也不知道为什么错了。Andrew 的核心洞察是：Agent 工程的首要目标不是最大化成功率，而是最小化不可预测性。

**Q: 为什么不要让 Agent 自己审查自己？**

A: Ash 的回答直击要害：开箱即用的 Claude 天然有宽容偏差和迎合倾向。当你让同一个模型既生成代码又审查代码，它倾向于给自己的输出打高分——不是因为代码真的没问题，而是因为模型在生成时已经建立了「这样做是对的」的内部表征，评估时会延续这个表征。更隐蔽的问题是：生成过程中模型做出了大量的隐含假设和权衡取舍，评估时很难跳出这些假设去发现它们本身可能就是错误的。Ash 的解决方案是借鉴 GAN 的对抗式设计：让 evaluator 成为一个真正独立的角色——它不参与生成、不理解生成者的内部决策过程、只从最终产物的行为和表现来判断质量。这种独立性是打破循环论证的关键——evaluator 不会「理解」为什么生成者做了某个选择，它只会测试这个选择在实际运行中是否 work。

**Q: generator 和 evaluator 具体怎么协作？contract 机制如何运作？**

A: 协作流程分为三个阶段。第一阶段是协商——generator 和 evaluator 在任务开始前共同定义一份 contract：明确「什么叫完成」，列出所有需要满足的具体标准。这些标准不是模糊的「做一个好应用」，而是可验证的条目，比如「精灵编辑器的删除功能必须支持多选」「路由必须正确处理无效 URL」。第二阶段是执行——generator 负责构建，evaluator 在构建完成后用 Playwright 打开真实页面，执行点击、输入、导航等用户操作，测试所有 contract 条目。第三阶段是反馈——evaluator 不给出笼统的评价，而是给出具体的、逐条的批评：「第 14 条未通过，原因如下，这是截图。」然后 generator 根据这些具体批评进行修复。Ash 强调，contract 是 Ralph loop 缺少的关键创新——它把「什么叫完成」从生成者的内部判断变成了两个独立角色之间的外部协议。

**Q: 27 条 contract 标准是怎么来的？为什么这个数字重要？**

A: 在 Retro Forge（复古游戏制作器）案例中，Ash 的团队经过多轮迭代，最终形成了 27 条 contract 标准。这些标准不是一次性设计出来的，而是在 generator 反复跑偏、evaluator 反复漏检的过程中，通过读 traces 发现盲区、然后逐条增加的。第 14 条关于精灵编辑器删除逻辑的标准，就是因为 evaluator 之前没有测试多选删除场景，导致一个生产环境的 bug 逃逸了。27 条这个数字本身不重要——重要的是它反映了一种工程方法：标准的粒度决定了批评的质量。如果只有 3 条模糊标准（「功能完整」「UI 好看」「没有 bug」），evaluator 的批评也会是模糊的，对 generator 没有指导价值。27 条具体标准意味着每一条都是可独立验证的，批评变成了可执行的指令。Ash 的隐含判断是：Agent 工程中的大量问题不是模型能力不够，而是评判标准不够细。

**Q: 主观质量——比如「设计好不好看」——怎么用模型来评分？**

A: Ash 的方法论是「拆解 + 校准」。第一步是拆维度——把「设计好不好看」这个整体判断拆成设计感（layout、配色、留白）、原创性（是否只是模板化的 Bootstrap 风格）、工艺感（动画流畅度、交互细节、loading 状态处理）、功能性（信息架构是否清晰、用户是否能快速完成任务）。第二步是校准——给每个维度提供 few-shot 示例，明确告诉 evaluator「这是 9 分的设计」「这是 3 分的设计」，用具体的视觉描述让模型建立评判参照系。第三步是结构化输出——让 evaluator 给每个维度单独打分并附具体理由，而不是给出一个笼统的美学评价。Ash 坦承这不完美——human taste 的某些维度仍然难以被完全捕获——但在实践中这套方法已经显著提升了 evaluator 的批评质量和 generator 的输出质量。

**Q: 模型变强后，harness 应该做加法还是减法？**

A: Ash 的回答是做减法。他用自己的开发经验举例：随着模型能力的跃迁，团队发现一些过去必需的 harness 组件变得多余——比如某些 context reset 步骤，过去是因为模型容易在长上下文中迷失方向而设计的，新版模型有了更好的上下文管理能力后，这些 reset 反而打断了 Agent 的连贯思考。再比如 evaluator 的运行节奏：早期每个 sprint 都要跑一次评估，因为模型容易在迭代中跑偏；新版模型的稳定性提升后，评估可以推迟到生成完成后一次性进行，节省了大量 token 和时间。但 Ash 同时警示不要走向另一个极端——不是所有 harness 都可以删掉。planner、generator、evaluator 三个角色的核心架构仍然保留，说明某些结构性的约束在任何模型能力下都是必要的。harness 设计的核心能力不是建得足够多，而是知道什么时候该删什么。

**Q: 「读 traces」具体怎么读？为什么 Anthropic 仍然大量手动读而不是用工具分析？**

A: Ash 描述的过程出人意料地朴素：把 Agent 的完整原始输出从日志里拉出来，从头到尾一行行读。看模型在每个决策节点看到了什么上下文、做出了什么推理、得出了什么结论。重点找两个东西：一是模型判断和人类判断的偏差——模型说「这个功能已经完成了」，但人类测试后发现还有 bug，回溯 traces 看模型为什么错判；二是上下文的漂移——在哪一个节点模型开始偏离初始目标，是什么信息或推理步骤导致了漂移。Ash 有时会用另一个 Agent 来 grep 日志辅助查找模式，但最终的解读和判断仍然依赖人类的仔细阅读。原因在于：当前的工具化 trace 分析（自动摘要、异常检测）仍然会丢失大量微妙信息——比如模型推理中隐含的假设、看似合理但实际错误的因果链条、上下文漂移的渐进过程。这些信息需要人类的细腻理解才能捕捉，而正是这些微妙之处区分了好 Agent 和差 Agent。

**Q: 这套模式（generator-evaluator）更适合 greenfield 项目还是已有生产项目？**

A: Ash 的回答是两者都适用，但切入方式不同。Greenfield 项目的优势是没有历史包袱——你可以从一开始就建立 contract、设置 evaluator、设计面包屑机制。Ash 分享的 Retro Forge 和 DAW 音乐应用案例都是 greenfield。对于已有生产项目，策略是渐进式的：先引入 evaluator 对现有功能做一轮全面评估，生成一份「当前状态报告」，然后在新功能开发中逐步建立 contract 标准。Ash 特别提醒：在已有项目中，evaluator 抓出的第一批问题可能会多得让人沮丧——这不是 harness 的问题，而是之前缺乏独立评估机制的自然结果。关键是不要试图一次性修复所有问题，而是从下一个 sprint 开始，让每个新功能都经过 generator-evaluator-contract 的完整流程。

**Q: human-in-the-loop 应该放在什么位置？像 sprint review 一样吗？**

A: Ash 将 human-in-the-loop 定位为类似于敏捷开发中的 sprint review。人的介入不应该在每一次 generator-evaluator 迭代之间——那会破坏 Agent 的自主运行效率。人的最佳介入点是在 evaluator 完成评估后、generator 开始新一轮修复前：人类审阅 evaluator 的批评报告，判断批评是否合理、优先级是否正确、是否需要调整 contract 标准。这种定位充分利用了人类的比较优势——模型擅长执行和模式匹配，人类擅长做价值判断和优先级决策。Ash 还提到，如果条件允许，让一个 PM 角色在循环中控制范围蔓延（scope creep）也是有价值的——因为 Agent 在自主运行时可能会倾向于不断扩大功能范围，PM 的介入可以帮助保持焦点。但 Ash 强调：人类的介入频率不宜过高，否则 Agent 的长时间自主运行就失去了意义——理想状态是像 sprint review 一样，每完成一个有意义的功能增量时做一次 checkpoint。

**Q: Agent teams 和 generator-critic 是什么关系？竞争还是互补？**

A: Andrew 认为是互补而非竞争。Agent teams 解决的是「谁做什么」的劳动力分工问题——把大任务分拆给不同的子 Agent，每个子 Agent 负责一个子领域。generator-critic 解决的是「做得好不好」的质量控制问题——通过独立评估角色来确保输出质量。两者可以而且应该组合使用：用 Agent teams 做任务分发和并行执行，然后在每个子 Agent 的输出上应用 generator-critic 模式做质量把关。Andrew 提到 Anthropic 内部已经在这样组合使用——一个 planner Agent 拆解任务分发给多个子 Agent（Agent teams），每个子 Agent 的输出在被汇总前先经过独立的 evaluator 审查（generator-critic）。这个组合的价值在于它同时解决了 Agent 系统的两个核心问题：规模和可靠性。

**Q: evaluator 应不应该看到 generator 的执行轨迹？**

A: Ash 的回答是：不应该看到完整的执行轨迹，只应该看到最终产物。这个选择有明确的理论依据——如果 evaluator 看到了 generator 的完整推理过程（为什么选这个库、为什么这样设计架构、中间遇到了什么困难），它就会不自觉地被 generator 的推理框架所影响，倾向于在同样的假设基础上做评判，从而失去独立批评的能力。Ash 将 evaluator 的行为模式类比为「黑盒测试」——一个好的 QA 不应该问开发者「你是怎么实现的」，而应该只看最终产品是否满足 spec。但 Ash 也承认这里有一个权衡：如果 evaluator 完全看不到过程信息，有时会产生不合理的批评——比如批评一个实际上是合理的架构选择。团队目前的实践是在 contract 中明确定义哪些过程信息 evaluator 可以访问、哪些不能，这是一条仍在探索中的边界。

**Q: 从一次性 demo 到长期产品，中间差了什么？**

A: Ash 的回答聚焦于持久化。一次性 demo 的 Agent 系统运行在「无状态」模式下——跑完就没了，下次启动从零开始。长期产品需要 Agent 在文件系统中建立完整的「外部记忆」：JSON 状态文件记录当前进度和待完成项，时间戳日志记录每次运行的关键事件，bug 记录追踪已知问题和修复历史，文件结构说明让 Agent 重新启动后能快速理解项目布局。Ash 特别强调这些「面包屑」的格式应该同时对人可读和对模型可读——JSON 格式适合模型解析，但需要配以足够的注释和说明让人类也能快速理解。这项工作看似琐碎，但它决定了 Agent 系统能否从「一次性运行」跨越到「持续演化」——没有外部记忆的 Agent 永远只能是一次性的，而有了外部记忆的 Agent 可以在每次运行的基础上积累进步。

## 三、备忘

- 本期克隆自 AI Engineer Conference 的技术分享《Build Agents That Run for Hours (Without Losing the Plot)》。AI Engineer Conference 是 AI 工程领域最具实操性的会议之一，本期分享来自 Anthropic 应用 AI 团队的两位一线工程师，内容密度极高——几乎每一段都包含可落地的工程判断而非空泛的原则表述。

- 「Ralph loop」这个名字值得留意。Andrew 没有在分享中详细解释名字的由来（show notes 中也未提及），但它已经成为 Anthropic 内部对「循环调用 Claude Code，每轮开新上下文，用确定退出条件控制风险」这一模式的代称。对开发者来说，Ralph loop 的核心启示不在于名字，而在于它背后的工程哲学：承认模型的不确定性，然后通过结构而非祈祷来控制它。

- 「确定性差」这个表述有被误读的风险。它不是在鼓吹追求失败，而是在说：在设计 Agent 系统时，可预测的失败模式比不可预测的成功更有工程价值。这和 Dan Shipper 在 #554 中提到的「拥抱 AI 的不确定性」形成了有趣的呼应——两位嘉宾从不同角度都在说同一件事：AI 系统的工程设计思路应该从「消除不确定性」转向「容纳不确定性」。

- generator-evaluator 的 GAN 类比是一个强有力的认知框架。GAN 之所以有效，不是因为 generator 或 discriminator 单独有多强，而是因为它们之间的对抗推动双方同时进步。Ash 将这一模式移植到 Agent 系统的含义是：evaluator 的质量和 generator 的质量是共生关系——evaluator 越严格，generator 被迫变得更好；generator 变得更好后，evaluator 又需要升级标准。这是一个螺旋上升的机制，而驱动的燃料就是独立评估。

- 27 条 contract 标准的案例是本期最具复制价值的方法论。它暗示了一个通用的工程实践：当你发现 Agent 跑偏时，不要先改 prompt 或加架构，而是先检查你的 success criteria 是否足够细。大多数情况下，「Agent 做得不好」实际上是「我们没告诉 Agent 什么叫做得好」。

- 「标准模糊，批评就会模糊」是本期最值得贴在显示器上的金句。它适用于 Agent 工程的几乎每一个环节——从 prompt 设计到 evaluator 配置到 human review。如果你发现自己对 Agent 的输出不满意却又说不清哪里不满意，问题很可能不在 Agent 而在你的标准。

- Ash 关于「读 traces」的回答——完整地、一行行地读原始输出——可能会让一些开发者失望，因为它听起来太「手工」了。但这恰恰是本期最有价值的诚实时刻之一：当前 Agent 工程的最前沿，在某些方面仍然依赖于最古老的方法——仔细阅读。工具可以辅助（用 Agent 来 grep 日志），但不能替代人类的细腻理解。这对那些在寻找「自动化一切」银弹的团队是一个重要的提醒。

- DAW 音乐应用的案例（更少轮次、更低成本、更完整应用）与 Retro Forge 案例形成了有趣的对照。两个案例使用相同的基础架构（planner-generator-evaluator），但 Retro Forge 需要更多轮次的迭代和更密集的 evaluator 介入，而 DAW 应用在更新的模型上以更少的轮次达到了更好的效果。这个对比精确地说明了「模型前沿在移动」的含义——同样的 harness 设计，在不同模型能力下的表现截然不同。

- human-in-the-loop 被定位为 sprint review 级别而非 micro-management 级别，是一个重要的设计决策。它体现了一个更深层的判断：Agent 自主运行的价值在于连续性，如果人介入得太频繁，就退回到了传统的「人写代码、AI 辅助」模式。sprint review 级别的介入频率——每完成一个有意义的功能增量时——在保持自主性的同时提供了必要的质量锚点。

- 「面包屑」机制——JSON 状态文件、时间戳日志、bug 记录、文件结构说明——是实现 Agent 长期运行最被低估的工程实践之一。它不性感，不涉及任何前沿技术，但它决定了 Agent 能否在上次运行的基础上继续工作还是每次都要从零开始。对于任何想要构建生产级 Agent 系统的团队，这是优先级最高的基础设施投入。

- 本期嘉宾的工程态度值得注意：Ash 和 Andrew 在整场分享中没有一次提到「AGI」「超级智能」或任何宏大叙事。他们的全部注意力都在工程实践上——怎么设计 harness、怎么读 traces、怎么定义标准。在一个充满 hype 的领域，这种脚踏实地的工程师精神本身就是一种稀缺的价值信号。
