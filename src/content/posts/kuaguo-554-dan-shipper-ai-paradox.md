## 一、核心观点

1. **AI 悖论：自动化越多，人要做的工作反而越多**：Dan Shipper 提出一个核心反直觉观察——尽管 Benchmark 显示模型能自主完成 17 小时以上的任务，但 Every 团队人数一年翻倍（从 15 到 30 人），他自己的工作量也更大了。原因是每增加一项自动化，就需要一个人去"盯"它确保正常运转。自动化不是让人闲下来，而是让人去做更高层次的管理和判断。

2. **每个 Agent 都需要一个"在乎它的人"**：Dan 从实践中得出结论——AI agent 要真正有用，需要与某个具体的人建立连接，这个人要持续关注它在做什么、确保做对了、对大家有用。一旦切断这个连接，agent 立刻就没那么有用。这就是为什么个人 agent 的热潮会退潮，转向全公司一个 super agent 的模式。

3. **未来工作方式分两条路：Slack 里的 Super Agent + 电脑上的 Codex/Claude Code**：第一条路是公司有一个超级 agent（如 Shopify/Ramp），所有人通过 Slack 跟它对话委派任务——先从顶层通用 agent 开始，再下沉为各团队专门 agent。第二条路是 Codex 或 Claude Code 成为承载一切工作的"操作系统"，在里面完成文档、邮件、研究等所有工作。

4. **SaaS 末日论是错的，Agent 增加而非消灭 SaaS 用量**：Dan 明确表示"如果是现在我会买 SaaS 股票"。因为 agent 会以极高频率使用 SaaS 产品（GitHub 就是因为 agent 用量导致用户数指数级飙升），Every 的 SaaS 支出同比仍在增长。SaaS 公司的挑战不是生存，而是同时为人和 agent 构建产品。

5. **CLI 时代已经结束，Agent 的最终界面是 GUI**：Cloud Code 刚火时很多人认为关键是 CLI，但 Dan 判断 CLI 阶段已经"快进跑完了"。把 agent 放进真正的 GUI（如 Codex Desktop App）里工作更舒适，尤其是对非程序员。当初发明 GUI 是有原因的——内置浏览器 + agent 并行工作的组合才是未来。

6. **PM 和全栈设计师是 AI 时代的最大赢家**：Dan 用 Every 内部案例说明，PM 背景的人借助 AI coding tools 发版速度比大多数工程师都快，因为他们有极强的产品感觉和用户敏感度。全栈设计师不再需要把设计交接给工程师，自己就能提 pull request 实现——创造力在 slop（低质内容）泛滥的时代更有价值。

7. **模型让"昨天的人类能力"变便宜，人类永远往上走一层**：模型的核心机制是把过去的数据吸收后以低成本部署出来。于是这些能力被商品化、变得不值钱。人类的任务是站到这些"被冻结的昨日能力"上面，去创造新的、更有意思的东西。Benchmark 可以刷满，但人总能找到更高一层的框架来审视问题。

8. **"Ride the Models" 是唯一不会输的策略**：不要害怕新模型，而是每有新模型发布就去"翻那些石头"——用它来做你在意的事，看看之前做不到的事现在能不能做了。保持好奇和玩心，让模型扩展你的能力边界。AI 的前沿不在旧金山，而在任何一个"AI 遇到真实的人做真实的事"的地方。

## 二、关键问答

**Q1: Dan Shipper 为什么看好个人 agent 后来又完全反过来？**
A: 最初 OpenClaw 出来时 Everyone 都开始用，Dan 非常相信"每个人都有一个 agent"的未来。但很快发现 agent 需要太多精力维护——它老是坏、需要摸索、需要 sh into 服务器。大多数人没法花这个时间。推动这件事的底层原因是 agent 需要有人"在乎它"，所以转向了一个部署工程师维护全公司 agent 的模式。

**Q2: Super Agent 的架构会是什么样？**
A: 先从顶层最通用的 agent 开始，公司里所有人都用它。随着大家更熟悉 use case，再下沉变为更专门的团队 agent。模型变得更独立后这种模式会继续往下渗透到个人层面。但本质上"平行组织架构图"的完整形态还需要时间。

**Q3: 为什么 Codex/Claude Code 会变成操作系统？**
A: Dan 现在所有工作都在 Codex 里——内置浏览器打开 Proof 写文档，Codex 在旁边并行工作看到一切；用 Codex+Core（email agent）处理邮件到 inbox zero；找候选人时 Codex 自动 research 并找到完美人选。当 AI agent 能访问你能访问的一切、看到你看到的一切，"把浏览器放进 agent"比"把 agent 放进浏览器"更有价值。

**Q4: SaaS 如何在 agent 时代生存？**
A: 为人和 agent 的"协作"设计产品——双方同时在同一个工作成果上做事。需要审批机制、inbox 总结、日志、快速回滚能力。不需要做太多 AI 界面，因为用户会带着自己的 AI（用自己的 token）来用你的产品。这反而保住了 SaaS 公司的利润率，因为不用为 token 成本买单。

**Q5: 为什么 Dan 说"自动化是个谎言"？**
A: 每当你把某件事自动化后，上面还需要一个人盯着确认它确实在正常工作。人类和 AI 一起工作的方式越来越像"当经理"——经理不是在沙滩上躺着，而是不断跟员工沟通、搞清楚怎么让事情运转更好。做模型的经理同样需要大量时间和注意力。

**Q6: Senior Engineer Benchmark 说明了什么？**
A: Dan 用自己 Vibe Coding 做出来的产品在线上不断崩溃的经历做了一个 benchmark——让模型从第一性原理重写代码。GPT 5.5 得分 62/100，之前所有模型只有 30 分，而人类高级工程师能拿 80-90 分。关键发现：GPT 5.5 是唯一有"主动性"的模型——敢直接拆掉旧代码重写。但模型永远会遵循"去修这些 issue"的指令，而真正的人类会先说"这代码就是一坨屎，必须重写"。

**Q7: AI 不会自己使用自己，Benchmark 有根本缺陷？**
A: Dan 指出任何真实使用场景里，AI 旁边总会有一个人离它很近，确保它在正常工作。Benchmark 分数上涨是因为问题已被框定好、说清楚、能打分——但很多真实人类工作只有被写下来之后才能被打分，而"先想清楚怎么 prompt 它"这个动作本身无法被衡量。所以即使 Benchmark 被刷满，也不等于完全替代了人类。

**Q8: 为什么创造力在 AI 时代会越来越重要？**
A: 模型让制造内容变得极其便宜，于是到处都是 Landing Page 和低质推文。因为大家用同样的模型、同样的默认方式，产出看起来都一样，变成了商品化的 slop。创造力——做出看起来非常不一样的东西——的价值只会越来越高，这是从 slop 海洋中脱颖而出的唯一方式。

**Q9: 哪些岗位变化最小？**
A: 目前销售（尤其是当面沟通部分）变化最小，因为销售太依赖人际交流。AI 主要在漏斗顶部（找线索、资格筛选）帮忙，但真正的销售对话还没发生根本变化。Dan 同时指出 CEO 和投资人"表面看起来"可以选择不用 AI，但一家公司在 AI 上能走多远取决于 CEO 在 AI 上走多远——这件事不能外包。

**Q10: AI 生成的文档和邮件会变成常态吗？**
A: 会，而且人们最终会接受甚至喜欢。Dan 举例 Every 季度规划完全由 Notion agent 完成——agent 跟每个人对话了解目标，生成高质量战略报告。对 AI 生成内容的排斥会消失，因为大多数人真的不擅长写战略文档和邮件。底线是：发送的人必须对每一行负责、知道里面写了什么。

**Q11: "两个 Agent 比一个更好"是什么意思？**
A: 当用户通过 Codex 作为主界面去访问另一个 agent（如 SaaS 应用）时，Codex 能提供远超用户手动输入的上下文信息（"这是我最近和 Dan 一直在做的所有事情"），两个 agent 可以来回沟通细节。这创造了一种非常定制化的体验——用户只需说"去修一下"，Codex 就会去跟那个应用沟通。

**Q12: 公司用 AI 后更多人提交 Pull Request 意味着什么？**
A: 非技术人员（咨询、运营、编辑）都在提 PR。过去构建东西很难，现在很容易，瓶颈从"能不能做出来"变成了"跟已做出来的其他东西放在一起是否合理、是否连贯"。这需要技术人员（architecture 型）负责以统一连贯的方式筛选和整合这些新代码。

**Q13: 如何看待"就业末日"的担忧？**
A: Dan 认为所谓 AI 导致的就业末日不太成立。很多公司重组是因为之前招太多人，AI 只是一个方便的借口。新模型让能力变得便宜后，会刺激更多需求——人人突然像工程师了，但对工程师的需求反而更大（需要他们判断哪些 slop 应该进 codebase）。工作岗位不会消失，但工作方式一定会改变。

**Q14: 普通人如何"Ride the Models"？**
A: 把模型用在你正在做的任何事情里。新模型出来就去试用、去"翻石头"（之前做不到的事现在能不能做了）。关键是保持好奇和玩心。模型一发布几乎马上就能用上——不需要在旧金山，你也可以是世界上最早发现它新用途的人。
