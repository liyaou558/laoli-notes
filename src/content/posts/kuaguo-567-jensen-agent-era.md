## 一、核心观点

### 1. Agentic AI 不是下一代「内容生成」，而是一套完整的任务执行系统

Jensen 将 Agentic AI 定义为四个组件的协同：Model（大脑，负责理解与推理）、Harness（推理与规划循环，决定下一步做什么）、Memory（短期与长期记忆，让 Agent 从经验中学习）、Tools & Runtime（执行层，调用 API、操作数据库、运行代码）。这不再是「提问-回答」的对话模式，而是「观察-推理-规划-执行-验证」的完整闭环。传统 AI 被问「这段代码对吗」，Agentic AI 被要求「找到 bug，修复它，写测试，跑 CI，部署到 staging」。Jensen 的金句直指核心：「下一波 AI 是 agentic AI。agentic AI 已经到来了。」这不是预测，是他的宣告。整个 GTC 的硬件和软件发布都在为这个宣告服务——NVIDIA 正在把自己从训练基础设施公司重新定义为 Agent 执行基础设施公司。

### 2. 新经济学：「Compute 就是收入，Token 就是利润」——数据中心成为 AI 工厂

Jensen 提出了一个激进的经济学重定义：数据中心不再是存储和冷却服务器的成本中心，而是 AI 工厂——一种制造 token 的生产设施。就像工厂把原材料转化为产品，AI 工厂把电力转化为 token，token 再转化为收入。在这个框架下，compute（算力）不再是成本项，而是直接产生收入的引擎。这意味着 CFO 的思维方式需要从「如何控制云计算账单」转向「如何在算力投资上获得最高 ROI」。Jensen 将此表述得毫不含糊：「Compute 现在就是收入，Compute 就是利润。」当算力等于利润时，基础设施投资的边际逻辑被彻底颠覆——买更多 GPU 不再是为了压降单位成本，而是为了捕获更多收入。

### 3. Vera Rubin：不是单一芯片，而是为 Agent 时代设计的全系统超级计算机

Vera Rubin 被定义为「为 Agent 打造的超级计算机」，其核心设计理念是：Agent 推理不是单次大模型调用的简单放大，而是多步骤、多轮次、包含大量中间状态的复杂计算模式。因此 Vera Rubin 不是一颗新 GPU，而是一个多机架 pod 级全系统架构——产规模是 Grace Blackwell 的两倍。它采用全新的无电缆、无软管、无风扇机架设计，从物理层面重构了计算密度和能耗比。Vera Rubin NVL72 专门针对提示处理和多步 agent 推理进行优化——这意味着 NVIDIA 不再只为「一次前向传播」设计硬件，而是为「Agent 进行几十步推理循环」来设计整个系统拓扑。演进路线清晰：Hopper → Blackwell → Vera Rubin，每一步都在往「Agent-native computing」靠近。

### 4. Vera CPU 为「数十亿焦躁的 Agent」设计——传统 CPU 的延迟假设被彻底推翻

Jensen 提出了一个极具画面感的论点：过去我们设计 CPU 时假定的用户是「能忍受秒级延迟的人类」。人类点一个按钮，等一两秒，这是可以接受的。但 Agent 以纳秒为时间单位运转，它们在毫秒内完成数十次推理步骤，每一次等待都是对系统效率的浪费。Vera CPU 正是为这种「纳秒级焦躁」而设计——它的内存带宽、I/O 路径和调度机制都围绕「Agent 永远不会等人」这一前提重新架构。Jensen 的原话极具冲击力：「过去我们为人类创造 CPU，而人类只有十亿人。未来会有数十亿个 Agent。」这句话同时做了两个论断：Agent 的数量将远超人类，而它们对计算的要求将远超人类。这是对半导体产业需求逻辑的根本性重估。

### 5. 个人计算的 Agent 化：你的笔记本不再是一台电脑，而是通往 AI 工厂的门户

Jensen 描绘的未来个人计算设备不再是一个独立完成所有工作的机器，而是一个嵌入了 agent runtime 的终端——它负责理解用户意图、协调本地与云端资源、将复杂任务路由到 AI 工厂去执行。笔记本的角色从「计算设备」变为「AI 工厂的客户端」。这意味着未来的操作系统需要内置 agent runtime，就像今天的 OS 内置网络栈一样基础。这也意味着个人计算设备的硬件设计逻辑将从「更强的本地算力」转向「更好的 Agent 体验」——包括更高效的网络连接、更智能的电源管理（因为 Agent 可能在后台持续运行）、以及能支持本地小模型与云端大模型协同推理的异构计算架构。

### 6. AI Coder 不只是写代码——Agent 需要与整个企业基础设施深度集成

NVIDIA 演示了用自然语言生成代码、写测试、并完成部署的全流程，但 Jensen 强调的重点不是「Agent 能写代码」——这已经是旧闻。真正的关键挑战是：Agent 需要与 CI/CD 流水线、身份管理系统、日志平台和治理框架无缝集成。如果 Agent 生成的代码不能自动进入企业的安全审计流程，不能继承正确的权限控制，不能被现有的监控系统观测到，那么它就不是一个真正的「企业级 AI 工程师」，而只是一个 demo。这个洞察将「AI coding」从个人工具视角拉到了企业级系统视角——Agent 的价值不只是生成的代码质量，而是它能否成为企业软件交付流水线中一个可信、可控、可审计的环节。

### 7. Agent 计算是「终极分布式计算模型」——NVIDIA 从 GPU 公司到 AI 基础设施公司的转型逻辑

Jensen 将 Agent 计算定义为「终极分布式计算模型」：一个 Agent 可能需要动态调用数十个不同的工具和 API，跨越多个数据中心和云区域，在推理过程中实时决策下一步去哪里获取计算资源。这种模式天然是分布式的、非确定性的、以任务而非以资源为中心的。NVIDIA 的战略回应是 CUDAX——一个 Agent 工具集，让开发者能够用统一接口调度分布在全球 AI 工厂中的算力。同时 DSX 用数字孪生技术来设计和优化 AI 工厂的布局。1 吉瓦级别的 AI 工厂已在规划中。区域 AI 云提供商的兴起意味着算力不再集中在少数超大规模云——这既是地缘政治驱动的，也是 Agent 计算对低延迟边缘推理的需求驱动的。整个图景是：NVIDIA 在搭建一个遍布全球的、以 Agent 计算为设计目标的分布式计算基础设施网络。

## 二、关键问答

**Q: Jensen 说的「Agentic AI 已经到来」具体指什么？**

A: 指的是 AI 从被动回答问题的模式进入主动执行任务的模式。它不是等用户问一句答一句，而是收到一个目标后，自己拆解步骤、调用工具、验证结果、修正错误，直到完成任务。NVIDIA 在本次 GTC 发布的所有硬件和软件——Vera Rubin、Vera CPU、CUDAX——都是为这种新模式构建的基础设施。

**Q: Agent 的四组件模型（Model / Harness / Memory / Tools）各自做什么？**

A: Model 是大脑，负责理解输入和生成推理。Harness 是决策循环，规划步骤并决定下一步。Memory 分为短期（当前任务上下文）和长期（跨任务经验积累）。Tools & Runtime 是执行层——调用 API、读写文件、操作数据库、触发部署。四个组件必须协同工作，缺一个就不是真正的 Agent，只是一个被动的语言模型。

**Q: 「AI 工厂」和传统数据中心有什么区别？**

A: 传统数据中心是成本中心——存放服务器、消耗电力、产生运维费用。AI 工厂是收入引擎——它将电力转化为 token，token 直接产生商业价值。就像工厂把铁矿石变成汽车，AI 工厂把能量变成智能服务。这意味着基础设施的经济核算逻辑从「如何省电」变成了「如何在每瓦特上产出最多 token 和最多收入」。

**Q: Vera Rubin 的产规模是 Grace Blackwell 的两倍，这意味着什么？**

A: 意味着 NVIDIA 不是在迭代单颗芯片，而是在 scaling 整个系统的产出能力。Agent 推理不是单次调用——一次任务可能触发几十甚至上百次模型调用。Vera Rubin 的 pod 级架构就是为了让所有这些调用在系统内部高效完成，减少跨机架通信损耗，让多步推理的时间成本大幅下降。

**Q: Vera CPU 为什么说传统 CPU 是为「能忍受秒级延迟的人类」设计的？**

A: 人类点一个按钮后等一两秒完全能接受，所以传统 CPU 的调度、内存和 I/O 都按这个容忍度设计。但 Agent 以纳秒速度运转，它在做几十步推理时每一步都在等，累积延迟会严重影响吞吐。Vera CPU 从底层假设上推翻了这个设计——它假定用户是永远不会等人的 Agent，所有延迟预算都被压缩到纳秒级别。

**Q: 「未来会有数十亿个 Agent」这个判断凭什么？**

A: Jensen 的逻辑是：每个人类用户不会只用一个 Agent。你会有一个写代码的 Agent、一个管理日程的 Agent、一个监控投资的 Agent、一个处理邮件的 Agent——甚至每个企业内部的每个业务流程都会有一个或多个 Agent 在持续运行。人类只有约百亿，但 Agent 可以轻松达到千亿甚至万亿量级。每个 Agent 都需要计算资源，这就是 NVIDIA 押注的市场规模。

**Q: Agent 计算为什么被称作「终极分布式计算模型」？**

A: 因为 Agent 在执行一个任务时，下一步需要什么资源是无法提前确定的——它可能在推理过程中决定去调用某个 API、查询某个数据库、或者启动一个子 Agent 在另一个数据中心运行代码。这种动态资源路由天然是分布式的、以任务为中心的，而不是以预先分配的计算节点为中心的。

**Q: NVIDIA 的 CUDAX 是做什么的？**

A: CUDAX 是一个 Agent 工具集和运行时框架，让开发者能够用统一接口让 Agent 调用分布在全球 AI 工厂中的算力资源。它解决的核心问题是：Agent 在执行多步推理时，需要动态地获取计算资源、调用工具、访问数据——CUDAX 提供了 Agent-native 的资源调度层，而不需要开发者手动管理分布式计算的细节。

**Q: 演示中的 AI Coder 和普通代码生成有什么不同？**

A: 普通代码生成是「给我写一个排序函数」→ AI 输出代码。NVIDIA 演示的 AI Coder 是「把这个功能从需求到部署全做完」→ Agent 生成代码、写测试、跑 CI、处理失败、修复问题、最终部署到 staging。它展示的是 Agent 完整参与软件交付流水线的能力，而不只是生成一段孤立的代码片段。

**Q: Agent 与企业工具链集成为什么被 Jensen 单独强调？**

A: 因为写代码只是工程的一小部分。真正的企业软件交付需要代码通过安全审计、继承正确的身份权限、被监控系统观测、符合治理框架。如果 Agent 生成的代码绕过了这些环节，它在 demo 里再惊艳也无法进入生产环境。Jensen 在强调：Agent 的价值边界不是代码质量，而是它能否成为可信的企业级交付节点。

**Q: 未来个人计算设备如何被 Agent 改变？**

A: 笔记本不再是一个独立完成所有计算的设备，而是一个嵌入了 agent runtime 的智能终端。它负责理解你的意图、在本地运行轻量模型做意图解析、将复杂任务路由到云端的 AI 工厂执行。操作系统将内置 agent 能力，就像今天内置网络栈一样基础。硬件设计也将从「更强的本地算力」转向「更好的 Agent 体验」。

**Q: 1 吉瓦 AI 工厂是什么概念？区域 AI 云为什么在兴起？**

A: 1 吉瓦相当于一个大型核电站的发电量——NVIDIA 正在规划这种规模的专用 AI 计算设施，说明其对 Agent 时代算力需求的预估极为激进。区域 AI 云的兴起则是因为：一方面各国出于数据主权考虑希望算力本地化，另一方面 Agent 的实时推理需要低延迟——让计算靠近用户和 Agent 的所在地是刚需。

## 三、备忘

- Jensen 本次 GTC Taipei 的核心叙事线索：Agent 不只是软件范式，它是一场需要全栈硬件重构的计算革命。从芯片（Vera CPU）、到系统（Vera Rubin pod）、到网络（无电缆机架）、到数据中心（AI 工厂）、到软件层（CUDAX），每一层都在为 Agent 计算重新设计。
- 「Compute 就是收入，Token 就是利润」可能成为未来几年科技 CFO 的口头禅。这背后的财务含义是：GPU 采购从 CapEx 折旧逻辑变成了直接 Revenue Generation 逻辑，这会改变 CFO 如何评估 ROI 和批准预算。
- Vera Rubin 的无电缆无软管无风扇机架设计值得关注——这不仅是散热和可靠性的工程进步，更是大规模部署时运维成本的质变。当 AI 工厂达到吉瓦级别，传统机架的维护复杂度会变成瓶颈。
- 「数十亿个 Agent」这个数量级判断值得记住。如果成真，相当于每一个人类对应多个 Agent——这意味着 Agent 之间的通信、协作和资源竞争将成为新的基础设施问题。当前我们讨论的是「人类如何使用 Agent」，未来可能需要讨论「Agent 之间如何协作」。
- 区域 AI 云 + 数据主权 + Agent 低延迟需求的三角关系是一个被低估的趋势。它意味着全球算力格局不会像互联网那样被几家超大规模云完全垄断，而是可能出现更分散的区域性算力节点网络。
- Vera CPU 的「纳秒级焦躁」是一个很好的框架——它把 Agent 的设计约束从软件层推到了硅片层。这意味着未来芯片设计的分野不只是「训练 vs 推理」，还要加上「人类交互 vs Agent 自主运行」。
- 整场演讲的隐含信息：NVIDIA 的竞争对手已经不再是 AMD 或某个芯片公司，而是任何试图定义 Agent 时代计算架构的玩家——包括自研芯片的云厂商和可能从模型层向下渗透的 AI 公司。
