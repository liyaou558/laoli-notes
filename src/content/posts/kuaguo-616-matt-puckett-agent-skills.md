## 一、核心观点

### 1. "技能地狱"的本质不是技能太少，而是缺乏评判标准

Matt Puckett 指出，AI 工程领域正在经历新一轮"地狱"：从"教程地狱"到"框架地狱"，再到当前的"技能地狱"。开发者手头有大量 Agent 技能，但拼凑起来的实际效果远不如预期。根本原因不在于技能不够多，而是没有一个共享的评判标准、没有一个框架来审视技能并把它变得更好。就像没有代码审查标准的团队会积累技术债一样，没有技能质量标准的 Agent 会在生产环境中不断制造意外。走出技能地狱的关键第一步，就是承认"能跑"不等于"写得好"，然后建立一套系统性的检查清单。

### 2. 触发方式决定了一切：模型调用技能的隐藏代价是不可预测性

Matt 将技能的触发方式分为两种：用户调用（用户主动 @技能名）和模型调用（模型在推理过程中自行决定加载技能）。模型调用技能的机制是上下文指针指向 skill.md，这意味着每次模型自主加载技能都会消耗上下文窗口。但 Matt 认为真正的代价不是 token 消耗，而是不可预测性——"每次你有一个模型调用技能，基本上你就要付出不可预测性的代价"。你不知道模型什么时候会调用、会不会调用、调用了之后会不会遵循。相比之下，用户调用技能虽然增加了用户的认知负载，但消除了不确定性。Matt 的个人偏好明确倾向于用户调用，这也解释了为什么 MatPat Skills 与 superpowers 在设计哲学上走了完全不同的路。

### 3. 技能内部只有两个核心单元：步骤和参考

Matt 把技能的内部结构拆解为两个基本单元：步骤（告诉 Agent 按什么顺序做什么事）和参考材料（步骤执行过程中需要用到的知识、模板或规范）。他给出的实战方法是：从零写技能时先定步骤，再定参考，而不是反过来往文件里堆材料。以 2PRD 技能为例，步骤部分定义产品需求文档的生成流程，参考部分提供 PRD 模板和质量检查清单。这种"先步骤后参考"的顺序确保了技能始终以可执行为导向，而不是变成一篇知识库文章。

### 4. 让主 skill.md 尽可能小：识别分支，把资料藏在上下文指针后面

Matt 的第三条核心建议直指技能文件的规模问题：让主 skill.md 文件尽可能小。瘦身的技巧是识别技能的不同分支——以 domain modeling 技能为例，它可能包含 DDD 建模、事件风暴、CRC 卡片等多种子路径。如果把所有分支的详细指导和参考资料都塞进主文件，不仅浪费上下文窗口，还会让模型在无关信息中迷失。解决方案是把分支资料藏在上下文指针后面，只在模型实际进入某个分支时才加载对应的子文件。这相当于给技能做了"懒加载"。

### 5. 引导词：把大量含义压缩到极小空间的艺术

引导是 Matt 框架中最具实操性的技巧。Agent 不按你的想法做事，往往不是能力不够，而是你没有用对引导词。引导词的概念是"有些词能把一大堆意思压缩到很小的空间里"——比如对编码 Agent 说"用垂直切片的方式实现"，不需要解释什么是垂直切片，这个词本身就能唤起一整套架构决策和行为模式。验证引导词是否生效的方法是观察推理轨迹：如果 Agent 在推理中复述了引导词并据此调整了行为，说明引导词已经内化。如果不生效，改进方向是增加当前步骤的"基础工作量"——通过隐藏未来目标和后续步骤，迫使 Agent 在当前步骤投入更多精力。

### 6. 隐藏未来步骤是增加基础工作量的核心技巧

Matt 给出了一个反直觉但极其有效的技巧：把技能拆成独立步骤，一次只让 Agent 看到一个步骤。他以计划模式中的"问澄清性问题"为例：如果不加干预，Agent 在计划阶段永远问不够澄清性问题，总是急于跳到执行。解决方案不是反复强调"多问问题"，而是把整个流程拆开——计划阶段只暴露计划相关的指令，隐藏后续所有执行步骤，Agent 不知道下一步要做什么，自然会在当前步骤投入更多精力。通过隐藏未来的目标来增加当前步骤的基础工作量，这是引导 Agent 行为的底层杠杆。

### 7. 剪枝三模式与删除测试：精简技能的终极武器

Matt 将剪枝归纳为三种需要快速排查的失败模式。模式一：不要重复自己——如果同一段内容出现在多个地方，改一处忘了另一处就会产生不一致，必须保持单一事实来源。模式二：冗余沉积——多人协作时每个人都往技能里加"有用的东西"，久而久之文件变成垃圾场，这是组织级技能腐化的经典路径。模式三：无效操作——那些看起来有用但删掉后不影响 Agent 行为的内容。Matt 给出了终极检测手段——删除测试："如果你把那段话直接删掉会怎样？Agent 大概率还是会写出一份像样的提交信息。"删掉后行为不变的内容，就不应该留在技能里。

### 8. 四大维度构成闭环：触发决定入口，结构决定布局，引导决定行为，剪枝决定质量

Matt 将整个框架总结为四个依次检查的维度：第一检查触发（谁调用、什么时候调用），第二检查结构（是否识别了分支、是否把参考资料藏到指针后面），第三检查引导（文本是否已浓缩成引导词、是否隐藏了不必要的未来信息），第四检查剪枝（有没有重复、有没有沉积、有没有无效操作）。这四个维度构成一个完整的技能质量闭环，Matt 强调上手的最佳方式是直接使用他发布的 Writing Great Skills 技能——用 Agent 来改进 Agent 技能，本身就是这套方法论的最佳实践。

## 二、关键问答

**Q: 什么是"技能地狱"？它和"教程地狱""框架地狱"有什么关系？**

A: "技能地狱"是 Matt Puckett 对当前 AI 工程领域困境的命名。之前开发者经历过"教程地狱"（大量教程但学完不会用）和"框架地狱"（框架太多不知道选哪个），现在进入"技能地狱"——手头有大量 Agent 技能但缺乏评判标准，分不清好坏，拼凑出来的效果远不如预期。共同点是：输入过剩而判断力缺失。

**Q: 模型调用技能和用户调用技能的核心区别是什么？**

A: 用户调用技能是用户主动 @技能名来触发，模型调用技能是模型在推理过程中自行决定加载。技术原理上，模型调用通过上下文指针指向 skill.md 将技能内容注入上下文。核心区别在于控制权归属和可预测性：用户调用增加了用户的认知负载但消除了不确定性，模型调用减少了用户操作但引入了"模型什么时候调用、会不会遵循"的不可预测性风险。

**Q: Matt 为什么偏好用户调用技能？**

A: Matt 认为"每次你有一个模型调用技能，基本上你就要付出不可预测性的代价"。他偏好用户调用技能的根本原因是消除这种不可预测性——当用户明确指定使用某个技能时，Agent 的行为边界是清晰的，不会出现模型自行判断"这个场景该不该加载技能"带来的意外。这种设计哲学与 superpowers 形成了鲜明对比，代表了两种不同的调用哲学。

**Q: 从零开始写一个 Agent 技能的正确顺序是什么？**

A: Matt 给出的方法是先定步骤，再定参考。步骤部分定义 Agent 按什么顺序执行什么操作，参考部分提供执行过程中需要用到的模板、规范和知识。很多人容易犯的错误是反过来：先往文件里堆大量参考材料，最后随便写几行步骤，结果是技能变成了一篇知识库文章而非可执行的操作流程。

**Q: 如何让主 skill.md 文件尽可能小？**

A: 核心技巧是识别技能的不同分支，然后把分支的详细资料藏在上下文指针后面。以 domain modeling 为例，DDD 建模、事件风暴、CRC 卡片等子路径的详细指导不应该全塞在主文件里，而是做成独立的子文件，只在模型实际进入该分支时才加载。这相当于给技能做了"懒加载"，既节省了上下文窗口，也避免了无关信息干扰模型判断。

**Q: 什么是引导词？为什么它比长篇指令更有效？**

A: 引导词是把大量含义压缩到极小空间里的词汇或短语。比如对编码 Agent 说"用垂直切片的方式实现"，不需要解释什么是垂直切片——这个词本身就携带了一整套架构决策和行为模式。长篇指令的问题在于消耗上下文且容易被忽略，而引导词利用的是模型对特定概念已有的理解，效率更高、触发更稳定。

**Q: 如何验证一个引导词是否真正生效？**

A: 观察模型的推理轨迹。如果 Agent 在推理过程中复述了引导词，并根据引导词调整了后续行为，说明引导词已被内化。如果推理轨迹里完全没有引导词的痕迹、行为也没有变化，说明引导词不生效，需要更换或增加配套的上下文强化。

**Q: "隐藏未来步骤"为什么能增加当前步骤的基础工作量？**

A: 当 Agent 能看到后续步骤时，它会本能地为后续步骤预留精力，导致当前步骤投入不足。以计划模式为例，Agent 知道下一步就要执行，所以"问澄清性问题"永远做不够。解决方案是把技能拆成独立步骤，一次只暴露当前步骤的指令，Agent 不知道下一步是什么，就会被"困"在当前步骤上，自然投入更多精力。

**Q: 剪枝环节需要排查的三种失败模式分别是什么？**

A: 模式一是"不要重复自己"——同一段内容出现在多个地方会导致维护时改一处漏一处，必须保持单一事实来源。模式二是"冗余沉积"——多人协作时每个人都往技能里加内容，久而久之文件变成垃圾场。模式三是"无效操作"——看起来有用但删掉后不影响 Agent 行为的内容。

**Q: 什么是"删除测试"？如何用它精简技能？**

A: "删除测试"是 Matt 给出的精简技能的终极手段：删除技能中的某段内容，然后观察 Agent 的行为是否发生变化。如果删掉后 Agent 的行为和之前完全一样，说明那段内容是无效操作，不应该留在技能里。Matt 举例说："如果你把那段话直接删掉会怎样？Agent 大概率还是会写出一份像样的提交信息。"

**Q: 触发、结构、引导、剪枝四个维度之间是什么关系？**

A: 四个维度构成一个完整的检查闭环。触发决定技能由谁在什么条件下调用，是入口问题。结构决定技能内部的步骤和参考资料如何布局，是组织问题。引导决定如何用最少的文字让 Agent 精确执行意图，是行为控制问题。剪枝决定什么内容应该被删除，是质量保障问题。Matt 建议按顺序逐个检查：先触发、再结构、再引导、最后剪枝。

**Q: 上手这套方法论的最佳方式是什么？**

A: Matt 建议直接使用他发布的 Writing Great Skills 技能——这是一个用来改进 Agent 技能的 Agent 技能，本身就是这套方法论的最佳实践和自举演示。用 Agent 来优化 Agent 技能，既降低了学习曲线，也体现了"好技能"的设计理念在实践中的效果。

**Q: 多人协作维护技能集时最常见的腐化模式是什么？**

A: 冗余沉积。每个协作者都想为技能"增值"，不断往里添加自己认为有用的内容，但没有人负责删除。久而久之，技能文件从精炼的操作指南膨胀为杂乱的百科条目，上下文窗口被浪费在越来越多无关信息上，Agent 的执行质量反而下降。解决方案是建立剪枝纪律——每次添加内容时同步检查是否有可删除的部分。
